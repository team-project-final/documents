### 2.1 환경 변수 관리

#### 파일 구조

| 파일 | 용도 | Git |
|------|------|-----|
| `.env.example` | 변수 목록 + 설명 (값은 placeholder) | **커밋** |
| `.env.local` | 로컬 개발 실제 값 | `.gitignore` |
| `.env.dev` / `.env.staging` / `.env.prod` | 환경별 비밀 값 | CI/CD Secrets only |

#### 네이밍 규칙

- 형식: `UPPER_SNAKE_CASE`
- 접두사로 서비스/도메인 구분: `PLATFORM_`, `KNOWLEDGE_`, `LEARNING_`, `ENGAGEMENT_`
- 공통 변수: 접두사 없이 `DB_HOST`, `REDIS_URL` 등

#### Good 예제

```bash
# .env.example — 커밋 대상
# ============================================================
# Database
# ============================================================
DB_HOST=localhost
DB_PORT=5432
DB_NAME=synapse_dev
DB_USER=synapse
DB_PASSWORD=<your-password-here>

# ============================================================
# JWT
# ============================================================
JWT_SECRET=<generate-with: openssl rand -base64 64>
JWT_ACCESS_EXPIRY=900
JWT_REFRESH_EXPIRY=604800

# ============================================================
# External APIs
# ============================================================
OPENAI_API_KEY=<your-openai-key>
STRIPE_SECRET_KEY=<your-stripe-key>

# ============================================================
# Service-specific
# ============================================================
KNOWLEDGE_ES_INDEX_PREFIX=synapse_knowledge
LEARNING_AI_MODEL=claude-sonnet-4-5-20250514
```

#### Bad 예제

```java
// 하드코딩된 비밀 값 — 절대 금지
public class ApiConfig {
    private static final String JWT_SECRET = "my-super-secret-key-123"; // 하드코딩
    private static final String OPENAI_KEY = "sk-proj-abc123...";      // 유출 위험
}
```

```bash
# .env.local을 커밋 — 절대 금지
git add .env.local  # 실제 비밀 값이 레포에 노출
```

### 2.2 Git 워크플로우 요약

#### 브랜치 네이밍

```
feature/{서비스접두사}-{이슈번호}-{설명}
```

| 서비스 | 접두사 | 예시 |
|--------|--------|------|
| platform-svc | `PLAT` | `feature/PLAT-042-social-login` |
| engagement-svc | `ENG` | `feature/ENG-015-badge-system` |
| knowledge-svc | `KNOW` | `feature/KNOW-088-graph-search` |
| learning-svc (Card) | `LEARN-CARD` | `feature/LEARN-CARD-023-srs-algorithm` |
| learning-svc (AI) | `LEARN-AI` | `feature/LEARN-AI-007-rag-pipeline` |
| frontend | `FE` | `feature/FE-031-note-editor-toolbar` |

#### 커밋 메시지

```
{타입}({스코프}): {제목 — 50자 이내}

{본문 — "왜" 설명}
```

| 타입 | 의미 |
|------|------|
| `feat` | 새 기능 |
| `fix` | 버그 수정 |
| `refactor` | 리팩토링 (기능 변화 없음) |
| `docs` | 문서 변경 |
| `test` | 테스트 추가/수정 |
| `chore` | 빌드/CI/설정 변경 |
| `perf` | 성능 개선 |

#### PR 규칙

| 항목 | 규칙 |
|------|------|
| 크기 | 400줄 이하 권장 (초과 시 분할) |
| 승인 | 팀장 + 도메인 오너 **2인 승인** |
| 브랜치 수명 | 최대 5일 → 강제 PR |
| 보안 관련 | 이중 승인 필수 (팀장 + 보안 담당) |

#### Good 예제

```bash
# 브랜치명
git checkout -b feature/KNOW-042-graph-search-filter

# 커밋 메시지
git commit -m "feat(graph): 검색 필터 옵션 추가

사용자가 노드 유형(노트/카드/태그)별로 그래프 검색 결과를
필터링할 수 있도록 함. 기존 전체 검색은 결과가 너무 많아
원하는 노드를 찾기 어렵다는 피드백 반영."
```

#### Bad 예제

```bash
# 브랜치명 — 정보 없음
git checkout -b feature/new-feature    # 서비스/이슈 없음
git checkout -b fix-bug                # 무슨 버그?

# 커밋 메시지 — 의미 없음
git commit -m "update code"            # 무엇을 왜?
git commit -m "fix"                    # 타입만, 설명 없음
git commit -m "feat: 기능 추가"         # 스코프 없음, 설명 모호
```

#### Rate Limit

| 대상 | 제한 | 헤더 |
|------|------|------|
| 비인증 | 60 req/min | `X-RateLimit-Limit: 60` |
| 인증 | 600 req/min | `X-RateLimit-Remaining: 599` |
| LLM 엔드포인트 | 30 req/min | `X-RateLimit-Reset: 1620000000` |

#### 멱등성

- **POST** 요청에 `Idempotency-Key` 헤더 필수 (카드 생성, 결제 등)
- 동일 키로 재요청 시 기존 결과 반환 (중복 생성 방지)

#### Good 예제

```bash
# 올바른 API 호출
curl -X POST https://api.synapse.app/api/v1/cards \
  -H "Authorization: Bearer eyJ..." \
  -H "Idempotency-Key: client-uuid-123" \
  -H "Content-Type: application/json" \
  -d '{"front": "Spring AOP란?", "back": "관점 지향 프로그래밍..."}'
```

#### Bad 예제

```bash
# 멱등성 키 없는 POST — 중복 생성 위험
curl -X POST https://api.synapse.app/api/v1/cards \
  -H "Authorization: Bearer eyJ..." \
  -d '{"front": "...", "back": "..."}'  # Idempotency-Key 누락
```

```json
// 비표준 에러 응답 — 절대 금지
{
  "error": true,
  "msg": "not found"
}
```

### 2.4 코드 리뷰 요약

#### 체크리스트 6축

| 축 | 핵심 확인 항목 |
|----|---------------|
| **기능** | 요구사항 충족, 엣지 케이스 처리, 로직 정확성 |
| **보안** | SQL Injection, XSS, Rate Limit, 인증/인가 |
| **테넌트 격리** | DB 필터, RLS, 캐시 키에 tenantId, Kafka 이벤트에 tenantId |
| **성능** | N+1 쿼리, 불필요한 루프, 인덱스 활용, 캐시 적절성 |
| **가독성** | 네이밍, 함수 크기, 단일 책임, 주석 적절성 |
| **테스트** | 단위/통합 테스트 포함, 커버리지 기준 충족 |

#### 피드백 태그

| 태그 | 의미 | 예시 |
|------|------|------|
| `[MUST]` | 반드시 수정 (머지 차단) | `[MUST] tenant_id 필터 누락` |
| `[SHOULD]` | 강력 권장 (이번 PR에서) | `[SHOULD] N+1 쿼리 — fetch join 추가` |
| `[COULD]` | 다음에 해도 됨 | `[COULD] shared로 추출하면 재사용 가능` |
| `[QUESTION]` | 이해를 위한 질문 | `[QUESTION] 이 타임아웃 값의 근거?` |
| `[PRAISE]` | 칭찬 | `[PRAISE] 이 추상화 깔끔합니다` |
| `[NIT]` | 사소한 스타일 | `[NIT] typo: recieve → receive` |

#### SLA

| 항목 | 기한 |
|------|------|
| 첫 리뷰 | PR 생성 후 **24시간** 이내 |
| 재검토 (수정 후) | 수정 push 후 **48시간** 이내 |

#### Good 예제 (리뷰 코멘트)

```markdown
[MUST] `NoteRepository.findAll()` 에서 tenant_id 필터가 없습니다.
현재 RLS로 방어되지만, 애플리케이션 레이어에서도 반드시 필터링해야 합니다
(이중 방어 원칙).

제안: `findAllByTenantId(TenantContext.getCurrentTenantId())` 로 변경
```

#### Bad 예제 (리뷰 코멘트)

```markdown
이거 왜 이렇게 했어요?          // 태그 없음, 비구체적
이상한데요                      // 무엇이 이상한지, 대안 없음
LGTM                            // 체크리스트 확인 없이 승인
```

### 3.1 로깅 규칙

#### 로그 레벨 기준

| 레벨 | 사용 시점 | 예시 |
|------|-----------|------|
| `ERROR` | 즉시 대응 필요한 장애 | DB 연결 실패, 외부 API 타임아웃 |
| `WARN` | 잠재적 문제, 곧 조치 필요 | Rate Limit 80% 도달, 재시도 발생 |
| `INFO` | 비즈니스 이벤트 | 로그인, 결제, 카드 생성, 노트 공유 |
| `DEBUG` | 개발/디버깅용 (prod 비활성) | SQL 쿼리, 요청 payload, 중간 계산값 |

#### 구조화 로깅 포맷

모든 로그는 JSON 형태로 출력하여 ELK/Loki에서 검색 가능하게 한다.

```json
  "timestamp": "2026-05-11T10:30:00.123Z",
  "level": "INFO",
  "service": "knowledge-svc",
  "traceId": "abc-123-def",
  "tenantId": "tenant-001",
  "userId": "user-042",
  "message": "노트 생성 완료",
  "context": {
    "noteId": "note-uuid-088",
    "tags": ["java", "spring"]
  }
}
```

#### 민감정보 마스킹

| 정보 | 마스킹 방식 | 예시 |
|------|-------------|------|
| 이메일 | 앞 2자 + `***` + 도메인 | `us***@gmail.com` |
| 토큰 | 앞 8자만 | `eyJhbGci...` |
| 비밀번호 | **절대 로그 금지** | — |
| 전화번호 | 뒤 4자리 마스킹 | `010-1234-****` |
| 카드번호 | 뒤 4자리만 | `****-****-****-1234` |

#### Good 예제 (Java — Spring)

```java
// 구조화 로깅 + 민감정보 없음
@Service
@RequiredArgsConstructor
@Slf4j
public class NoteService {

    @Transactional
    public NoteResponse createNote(NoteCreateRequest request) {
        Note note = noteRepository.save(mapper.toEntity(request));
        
        log.info("노트 생성 완료 [noteId={}, tenantId={}, userId={}]",
            note.getId(),
            TenantContext.getCurrentTenantId(),
            SecurityContext.getCurrentUserId());
        
        return mapper.toResponse(note);
    }
}
```

### 3.2 보안 체크리스트

PR을 올리기 전 아래 항목을 **자가 점검**한다.

#### PR 전 보안 자가 점검표

```markdown
## 보안 점검 (PR 제출 전 확인)

### 인증/인가
- [ ] 인증이 필요한 엔드포인트에 JWT 검증이 있는가?
- [ ] 리소스 접근 시 소유자(tenant/user) 확인을 하는가?
- [ ] 권한 레벨 체크가 올바른가? (일반 사용자가 관리자 API 호출 불가)

### 입력 검증
- [ ] SQL 쿼리에 사용자 입력이 @Param 바인딩으로 처리되는가?
- [ ] HTML 출력 시 XSS 방어 (sanitize/escape) 가 적용되는가?
- [ ] 파일 업로드 시 확장자/MIME 타입/크기를 검증하는가?
- [ ] URL 입력 시 SSRF 방어 (허용 목록 체크) 가 있는가?

### 비밀 값
- [ ] 코드에 하드코딩된 시크릿/키/비밀번호가 없는가?
- [ ] 로그에 토큰/비밀번호/개인정보가 출력되지 않는가?

### 의존성
- [ ] 새로 추가한 라이브러리에 알려진 취약점이 없는가?

### CORS
- [ ] 허용 origin이 명시적 화이트리스트인가? (와일드카드 * 금지)
```

#### Good 예제 (SQL Injection 방어)

```java
// Parameterized Query — 안전
@Query("SELECT n FROM Note n WHERE n.tenantId = :tenantId AND n.title LIKE :keyword")
List<Note> searchByTitle(@Param("tenantId") String tenantId, 
                         @Param("keyword") String keyword);
```

```dart
// Flutter — 사용자 입력 escape
final sanitizedQuery = HtmlEscape().convert(userInput);
```

#### Bad 예제 (SQL Injection 취약)

```java
// 문자열 연결 — 취약
String query = "SELECT * FROM notes WHERE title = '" + userInput + "'";
entityManager.createNativeQuery(query).getResultList();
// userInput = "'; DROP TABLE notes; --" → SQL Injection 발생
```

#### Good 예제 (XSS 방어)

```java
// Spring — Content-Type 강제 + HTML escape
@GetMapping(produces = MediaType.APPLICATION_JSON_VALUE)
public ApiResponse<NoteResponse> getNote(@PathVariable String noteId) {
    // JSON 응답이므로 HTML context 없음 — 안전
    return ApiResponse.success(noteService.getNote(noteId));
}
```

#### Bad 예제 (XSS 취약)

```java
// 사용자 입력을 HTML에 직접 삽입
@GetMapping(produces = MediaType.TEXT_HTML_VALUE)
public String renderNote(@PathVariable String noteId) {
    Note note = noteService.getNote(noteId);
    return "<h1>" + note.getTitle() + "</h1>"; // XSS 취약
}
```

### 3.3 멀티테넌시 공통 규칙

#### 전체 흐름도

```
┌─────────────┐     ┌──────────────────┐     ┌────────────────────┐
│   Flutter    │     │  Spring Gateway  │     │   Spring Service   │
│  (Client)   │────▶│  JWT에서 추출     │────▶│  TenantContext     │
│             │     │  X-Tenant-Id     │     │  자동 주입          │
└─────────────┘     └──────────────────┘     └─────────┬──────────┘
                                                        │
                    ┌──────────────────┐     ┌──────────▼──────────┐
                    │   PostgreSQL     │◀────│    Repository       │
                    │   RLS 이중 방어   │     │  WHERE tenant_id=?  │
                    └──────────────────┘     └─────────────────────┘
```

#### Backend 규칙

| 규칙 | 설명 |
|------|------|
| TenantContext | 모든 요청에서 자동 주입 (Gateway → Filter → ScopedValue) |
| Repository | Hibernate `@TenantId`로 자동 필터링, 수동 WHERE 금지 |
| 캐시 키 | `tenant:{tenantId}:{domain}:{key}` 형식 강제 |
| Kafka 이벤트 | CloudEvents 확장 속성으로 `tenantid` 필수 포함 |
| Elasticsearch | 인덱스 패턴: `synapse_{domain}_{tenantId}` 또는 필터 쿼리 |

#### Frontend 규칙

| 규칙 | 설명 |
|------|------|
| 로그인 시 | JWT decode → tenantId 추출 → `tenantProvider` 설정 |
| API 요청 | Dio 인터셉터에서 `X-Tenant-Id` 헤더 자동 주입 |
| 로컬 캐시 | Hive box 키에 tenantId prefix 포함 |

#### Good 예제 (Backend)

```java
// Hibernate @TenantId — 자동 필터링
@Entity
@Table(name = "notes")
public class Note extends BaseEntity {
    @TenantId
    @Column(name = "tenant_id", nullable = false, updatable = false)
    private String tenantId;
    
    // 모든 쿼리에 자동으로 WHERE tenant_id = ? 추가됨
}

// Redis 캐시 키 — tenantId 포함
@Cacheable(key = "'tenant:' + T(TenantContext).getCurrentTenantId() + ':note:' + #noteId")
public NoteResponse getNote(String noteId) { /* ... */ }
```

#### Bad 예제 (Backend)

```java
// 수동 WHERE — 누락 위험
public List<Note> findAllNotes() {
    return noteRepository.findAll(); // tenant_id 필터 없음! 전체 테넌트 데이터 노출
}

// 캐시 키에 tenantId 누락 — 다른 테넌트 데이터 반환 위험
@Cacheable(key = "'note:' + #noteId") // tenantId 없음!
public NoteResponse getNote(String noteId) { /* ... */ }
```

#### Good 예제 (Frontend)

```dart
// Dio 인터셉터 — X-Tenant-Id 자동 주입
class TenantInterceptor extends Interceptor {
  final Ref ref;
  TenantInterceptor(this.ref);
  
  @override
  void onRequest(RequestOptions options, RequestInterceptorHandler handler) {
    final tenantId = ref.read(tenantProvider);
    if (tenantId != null) {
      options.headers['X-Tenant-Id'] = tenantId;
    }
    handler.next(options);
  }
}
```

#### Bad 예제 (Frontend)

```dart
// 매 요청마다 수동으로 헤더 추가 — 누락 위험
final response = await dio.get('/notes', 
  options: Options(headers: {'X-Tenant-Id': tenantId})); // 매번 수동 = 실수 가능
```

### 3.4 에러 코드 & 메시지 관리

#### 에러 코드 체계

형식: `{도메인}_{에러유형}`

| 도메인 | 코드 예시 | HTTP Status |
|--------|-----------|-------------|
| AUTH | `AUTH_TOKEN_EXPIRED` | 401 |
| AUTH | `AUTH_INSUFFICIENT_PERMISSION` | 403 |
| NOTE | `NOTE_NOT_FOUND` | 404 |
| NOTE | `NOTE_QUOTA_EXCEEDED` | 403 |
| CARD | `CARD_REVIEW_INTERVAL_INVALID` | 422 |
| GRAPH | `GRAPH_CYCLE_DETECTED` | 422 |
| BILLING | `BILLING_PAYMENT_FAILED` | 402 |
| COMMON | `VALIDATION_FAILED` | 400 |
| COMMON | `RATE_LIMIT_EXCEEDED` | 429 |
| COMMON | `INTERNAL_ERROR` | 500 |

#### Backend — ErrorCode Enum 패턴

```java
@Getter
@RequiredArgsConstructor
public enum ErrorCode {
    // Auth
    AUTH_TOKEN_EXPIRED(HttpStatus.UNAUTHORIZED, "인증 토큰이 만료되었습니다."),
    AUTH_INSUFFICIENT_PERMISSION(HttpStatus.FORBIDDEN, "권한이 부족합니다."),
    
    // Note
    NOTE_NOT_FOUND(HttpStatus.NOT_FOUND, "요청한 노트를 찾을 수 없습니다."),
    NOTE_QUOTA_EXCEEDED(HttpStatus.FORBIDDEN, "노트 생성 한도를 초과했습니다."),
    
    // Common
    VALIDATION_FAILED(HttpStatus.BAD_REQUEST, "입력값 검증에 실패했습니다."),
    RATE_LIMIT_EXCEEDED(HttpStatus.TOO_MANY_REQUESTS, "요청 횟수를 초과했습니다."),
    INTERNAL_ERROR(HttpStatus.INTERNAL_SERVER_ERROR, "서버 내부 오류가 발생했습니다.");
    
    private final HttpStatus httpStatus;
    private final String message;
}
```

#### Frontend — 에러 코드 매핑


### 3.5 문서화 규칙

#### 코드 주석 원칙

> **"왜(Why)"만 기술한다. "무엇(What)"은 코드 자체가 설명한다.**

#### Good 예제

```java
// SRS 간격 계산 시 최소값을 1일로 제한한다.
// 이유: easeFactor가 극단적으로 낮아지면 간격이 0 이하로 떨어져
//       무한 복습 루프에 빠지는 버그가 발생했음 (LEARN-CARD-019 참고)
int newInterval = Math.max(1, (int)(currentInterval * easeFactor));
```

```dart
// GoRouter redirect에서 온보딩 완료 여부를 별도로 체크한다.

// AUTH 상태만으로는 구분 불가하므로 별도 플래그 확인.
```
