- 대상: `documents/docs/final-deliverables/report-deck.html` + 신규 `발표대본_Q&A_SYNAPSE.md`

## 3. 슬라이드당 콘텐츠 구조

각 콘텐츠 슬라이드의 `<aside class="speaker-notes">`를 아래 형식으로 갱신한다(평문 + 줄바꿈, presenter 패널에서 가독):

```
[대본]
(구어체 3~5문장. 슬라이드 핵심 메시지를 자연스러운 발표 톤으로.)

── 예상 Q&A ──
Q1. (실무자 질문)
A1. (모범답변 2~4문장, 슬라이드 근거·ADR·실측치 인용)
Q2. … / A2. …
(3~5쌍)
```

- 기존 화자노트의 좋은 문장은 [대본]으로 흡수·다듬기. 정보 손실 없이 개선.
- 답변은 슬라이드에 실제 있는 근거만 인용(예: SLA P1 79.7ms, JaCoCo 80%, BACKWARD 호환, Redis rate-limit 1rps·burst 60). 슬라이드에 없는 수치는 단정하지 않는다.

## 4. 예상 Q&A 방향 (슬라이드별 예시)

실무자(백엔드/인프라/AI) 관점의 트레이드오프·운영 질문. 예:
- slide-5/6: "왜 단일 PG + 스키마 격리인가? 서비스별 물리 분리 대비 장단점은?" / "Kafka+Outbox로 가용성 결합을 제거했다는데, 정확히 무엇이 분리되나?"
- slide-15: "JWT를 게이트웨이에서 검증하고 서비스로 신원 전파 — 서비스가 토큰 재검증 안 하면 내부 위변조 위험은?"
- slide-17: "Claude/OpenAI 이중화 폴백 기준은? 비용·레이턴시 트레이드오프는?"
- slide-20: "BM25 × 임베딩 RRF 융합에서 가중치/k는 어떻게 정했나? 정확도 측정은?"
- slide-21: "rate-limit 1rps는 너무 낮지 않나? 실제 운영 기준은?" / "Outbox 무한 재시도 폭주 대비는?"
- slide-33: "Outbox 무한 재시도를 max-attempt+DLQ로 바꾸면 유실 위험은 어떻게 막나?"

> 답변은 슬라이드 근거 + 06장 개선점(이미 인지한 한계는 솔직히 인정 + 개선 방향)으로 구성.

## 6. 작업 방식

- 단일 소스(JSON 또는 파이썬 dict: slide_id → {script, qa:[{q,a}]})를 작성.
- 파이썬 스크립트로 (a) 덱 `report-deck.html`의 해당 슬라이드 `<aside class="speaker-notes">` 내용 교체(결정적), (b) `발표대본_Q&A_SYNAPSE.md` 생성.
- aside는 HTML 이스케이프 처리, 줄바꿈은 `\n`(presenter 패널이 평문 표시) 또는 `<br>` 중 패널 렌더 방식에 맞춤 — 구현 시 presNotes 렌더 코드 확인 후 결정.
