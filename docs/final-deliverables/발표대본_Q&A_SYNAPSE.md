# SYNAPSE 발표 대본 + 실무자 예상 Q&A

> report-deck.html 화자노트와 동일 소스. 콘텐츠 슬라이드 26장.

## slide-4

**[대본]**

오늘 소개할 SYNAPSE는 개인 지식 관리, 즉 PKM과 간격 반복 학습 SRS를 하나의 워크플로우로 통합한 멀티테넌트 학습 SaaS입니다. 그 사이를 LLM이 자동으로 이어줘서, 한마디로 옵시디언과 앤키, 그리고 RAG를 융합한 포지셔닝입니다. 기존 도구들은 노트와 복습 카드가 분리돼 수동으로 옮겨야 했고, 카드를 원본 노트와 연결하지 못해 맥락이 사라졌습니다. 거기에 카드 제작 노동, 관계 시각화 부재, 키워드뿐인 검색 한계까지, 이 다섯 가지 문제를 풀고자 SYNAPSE를 선정했습니다.

**예상 Q&A**

- **Q1.** PKM과 SRS를 굳이 한 플랫폼에 통합한 이유가 뭔가요? 각 영역의 전용 도구가 이미 성숙해 있는데요?
  - **A1.** 두 도구가 분리돼 있으면 노트를 카드로 수동 복사해야 하고 복습 카드가 원본 노트와 끊겨 맥락을 잃습니다. SYNAPSE는 이 워크플로우 단절과 맥락 상실을 해소하려고 카드가 원본 노트에 링크되도록 통합했습니다. 통합의 핵심 가치는 그 사이를 LLM이 자동으로 이어주는 데 있습니다.
- **Q2.** Obsidian, Anki 같은 기존 강자가 있는데 차별점이 실제로 사용자가 체감할 만큼 큰가요?
  - **A2.** 기존 도구는 PKM과 SRS가 분리돼 카드 제작이 수작업이고, 의미 검색이 없으며 관계 시각화도 약합니다. SYNAPSE는 노트에서 AI가 카드를 자동 생성하고, 위키링크 지식 그래프와 의미 기반 하이브리드 검색을 제공해 그 격차를 메웁니다. 다만 성숙한 전용 도구 대비 기능 깊이는 앞으로 더 채워야 할 부분입니다.
- **Q3.** 멀티테넌트 SaaS라고 했는데 학습 데이터의 테넌트 격리는 어떻게 보장하나요?
  - **A3.** 멀티테넌트 학습 SaaS로 설계했고 데이터는 소유자 기준으로 격리합니다. 노트와 카드 같은 자원은 소유자 검증을 통해 접근을 제한합니다. 구체적인 격리 메커니즘은 뒤의 인프라와 도메인 슬라이드에서 더 다룹니다.

## slide-5

**[대본]**

SYNAPSE의 차별점은 제품과 아키텍처 두 축으로 정리됩니다. 제품 면에서는 PKM과 SRS를 한 플랫폼에 담아 카드가 원본 노트에 링크되고, LLM이 노트를 분석해 카드를 자동 생성하며, 위키링크 기반 지식 그래프와 BM25·임베딩을 RRF로 융합한 하이브리드 검색을 제공합니다. 아키텍처 면에서는 토이 프로젝트가 아니라 의사결정을 ADR로 남겼습니다. 0001 MSA로 4개 도메인을 네트워크·배포로 강제 분리하고, 0002 Kafka에 Avro와 Outbox로 가용성 결합을 끊었으며, 0003 단일 PostgreSQL을 스키마로 격리해 7인 팀의 비용 현실과 권한 분리를 함께 잡았습니다.

**예상 Q&A**

- **Q1.** ADR 0001에서 MSA를 택했는데, 7인 팀에 마이크로서비스는 과한 오버엔지니어링 아닌가요?
  - **A1.** MSA를 택한 건 4개 도메인의 경계를 네트워크와 배포로 강제해 학습 목표인 현업 표준 아키텍처를 경험하기 위해서입니다. 대신 ADR 0003에서 단일 PostgreSQL을 스키마로만 격리해 7인 팀의 비용 현실과 권한 분리를 절충했습니다. 즉 컴퓨트는 분리하되 데이터 인프라 비용은 통제하는 균형점을 ADR로 기록했습니다.
- **Q2.** ADR 0002에서 Avro와 Outbox를 같이 쓰는데, 동기 호출 대비 무엇을 얻고 무엇을 잃나요?
  - **A2.** 동기 호출은 소비자 장애가 생산자까지 전파되는 가용성 결합을 만드는데, Kafka에 Outbox를 더해 그 결합을 끊었습니다. Avro와 BACKWARD 호환 정책으로 스키마 진화 시 기존 소비자가 깨지지 않게 했습니다. 대가는 이벤트 흐름의 운영 복잡도 증가지만, 도메인 간 독립 배포를 얻습니다.
- **Q3.** 단일 PG 스키마 격리는 결국 단일 장애점 아닌가요? 도메인별 DB 분리가 정석 아닌가요?
  - **A3.** 맞습니다, 단일 PG는 공유 인프라라 장애 반경이 넓다는 트레이드오프가 있습니다. 다만 ADR 0003에서 7인 팀의 운영·비용 현실을 우선해 스키마 격리와 권한 분리로 절충한 의사결정입니다. 규모가 커지면 도메인별 DB 분리가 다음 단계이며, MSA 경계를 이미 잡아둔 만큼 이행 비용은 낮습니다.
- **Q4.** 하이브리드 검색에서 RRF를 택한 이유는요? 가중합 같은 다른 융합 방식도 있는데요?
  - **A4.** RRF는 BM25 점수와 임베딩 유사도처럼 스케일이 다른 순위를 정규화 없이 순위 기반으로 합쳐 안정적으로 융합한다는 장점이 있습니다. 가중합은 점수 스케일 보정과 튜닝 부담이 큽니다. 다만 저희는 정확도 정량 측정인 골든셋이 아직 미완이라, 융합 품질의 정량 평가는 개선 과제로 남아 있습니다.

## slide-6

**[대본]**

시스템 구성을 한눈에 보면, Flutter 클라이언트가 Web과 iOS, Android를 커버하고 게이트웨이가 JWT를 엣지에서 검증하며 레이트 리밋을 겁니다. 그 뒤로 platform, knowledge, learning, engagement 네 개 도메인 서비스가 있고, 이들은 Kafka와 Avro 위에서 8개 토픽과 Outbox로 통신합니다. 데이터 인프라는 pgvector를 얹은 PostgreSQL 16.9, Redis 7.1, 그리고 한국어 형태소 Nori를 쓰는 Elasticsearch 9.2이고, Kubernetes 1.30과 ArgoCD GitOps로 dev·staging·prod를 운영합니다. 노트 한 장이 생성되면 AI 카드 생성, 복습 완료, XP와 레벨업, 알림까지 이벤트가 순환하고 그 전 과정이 감사 기록으로 남습니다.

**예상 Q&A**

- **Q1.** 게이트웨이가 단일 진입점인데 여기가 죽으면 전체 서비스가 멈추는 단일 장애점 아닌가요?
  - **A1.** 게이트웨이는 JWT 엣지 검증과 레이트 리밋을 담당하는 단일 진입점이라 가용성이 중요합니다. Kubernetes 위에서 운영하므로 복제와 재기동으로 가용성을 확보합니다. 다만 다중 인스턴스 운영과 자동 복구를 더 강화하는 건 운영 성숙도 차원의 다음 과제입니다.
- **Q2.** PostgreSQL, Redis, Elasticsearch까지 데이터 저장소가 셋인데 일관성은 어떻게 맞추나요?
  - **A2.** PostgreSQL이 정본 저장소이고 Elasticsearch는 검색용 파생 인덱스입니다. 노트 변경은 Outbox로 이벤트를 발행해 Kafka를 거쳐 ES에 비동기 동기화하므로 최종 일관성 모델입니다. Redis는 캐시·레이트 리밋 용도로, 정합성은 DB 폴백으로 보완합니다.
- **Q3.** 토픽이 8개라고 했는데 이벤트 순환에서 무한 루프나 폭주는 어떻게 막나요?
  - **A3.** 노트 생성에서 카드 생성, 복습 완료, XP·레벨업, 알림으로 이어지는 단방향 순환 구조라 사이클이 닫혀 있습니다. 소비자는 멱등 처리와 DLQ를 둬 중복·실패를 흡수합니다. 다만 Outbox 무한 재시도 문제는 인지하고 있어 max-attempt와 DLQ 상한으로 보완하는 게 개선 과제입니다.

## slide-7

**[대본]**

개발 환경은 현업 표준 도구체인으로 맞췄습니다. 백엔드는 Java 21에 Spring Boot 4.0과 Modulith를 ArchUnit으로 경계 검증하고, AI는 Python 3.12와 FastAPI에서 anthropic·openai SDK를 직접 호출하며 pgvector를 씁니다. 프론트는 Flutter와 Dart 단일 코드베이스에 Riverpod, go_router를 쓰고, 데이터는 TLS를 건 Kafka와 Schema Registry, Nori를 얹은 Elasticsearch입니다. 인프라는 K8s 1.30, Terraform, Kustomize, ArgoCD이고 CI는 GitHub Actions 재사용 워크플로우에 Testcontainers와 JaCoCo 80% 게이트를 걸었습니다. 활용은 Free·Pro·Team·Enterprise의 Freemium SaaS로 B2C와 B2B를 모두 노립니다.

**예상 Q&A**

- **Q1.** AI 서비스만 Python·FastAPI로 빼고 나머지는 Java·Spring인데, 폴리글랏의 운영 부담은 어떻게 감당하나요?
  - **A1.** AI 영역은 anthropic·openai SDK와 pgvector 생태계가 Python에서 가장 성숙해 FastAPI로 분리했습니다. 도메인이 MSA로 이미 경계가 나뉘어 있어 언어 분리가 배포·운영에 큰 추가 비용을 만들지 않습니다. 공통 계약은 Avro와 Schema Registry로 언어 중립적으로 맞춥니다.
- **Q2.** JaCoCo 80% 커버리지 게이트를 강제하면 의미 없는 테스트로 수치만 채우는 부작용은 없었나요?
  - **A2.** 커버리지 게이트는 최소 안전망이고 실제 검증은 Testcontainers 기반 통합 테스트와 E2E로 보강했습니다. ArchUnit으로 Modulith 경계까지 검증해 구조적 회귀도 잡습니다. 수치 자체보다 결함을 이슈에서 PR, 회귀 검증으로 닫는 사이클을 더 중시했습니다.
- **Q3.** Spring Boot 4.0은 매우 최신인데 안정성 리스크를 감수한 이유가 있나요?
  - **A3.** 현업 표준 레퍼런스를 학습 목표로 삼아 최신 스택을 의도적으로 채택했습니다. 실제로 Boot 4.0의 YAML 라우트 버그 같은 이슈를 겪었고 Java config 전환으로 우회했습니다. 최신 스택의 리스크를 직접 부딪혀 트러블슈팅한 경험 자체가 학습 성과의 일부입니다.

## slide-9

**[대본]**

7인 팀의 역할 분담입니다. 팀장 김민구가 게이트웨이와 인프라, 즉 K8s·Kafka·DB IaC와 ArgoCD CI/CD, staging 복구를 맡았고, 김해준은 platform에서 OAuth·JWT·MFA 인증과 Stripe 결제, FCM·이메일 알림, 감사로그, GDPR을 담당했습니다. 한승완은 engagement의 커뮤니티와 게이미피케이션, 김현지와 박은서는 knowledge를 둘로 나눠 각각 노트·그래프·검색 동기화와 비동기 청킹·하이브리드 검색·Modulith 경계를 맡았습니다. 조유지와 김나경은 learning을 카드와 AI로 나눠 SM-2 복습과 Claude 카드 생성·RAG를 구현했고, Flutter 프론트는 별도 오너 없이 각자 자기 도메인 화면을 공동 구현했습니다.

**예상 Q&A**

- **Q1.** knowledge와 learning을 각각 두 명이 나눠 맡았는데, 한 도메인 안에서 경계 충돌이나 통합 비용은 없었나요?
  - **A1.** knowledge는 노트·그래프 동기화와 청킹·검색으로, learning은 카드와 AI로 책임을 명확히 나눠 충돌을 줄였습니다. Modulith 경계를 ArchUnit으로 강제해 같은 도메인 안에서도 모듈 침범을 컴파일 단계에서 막았습니다. 통합 지점은 PR 크로스리뷰로 조율했습니다.
- **Q2.** Flutter 프론트에 전담 오너가 없으면 UI 일관성이나 품질 책임이 분산되지 않나요?
  - **A2.** 프론트는 각자 자기 도메인 화면을 공동 구현하는 방식이라 백엔드 담당자가 자기 기능의 UI까지 책임지는 구조입니다. 단일 코드베이스라 컨벤션을 공유하기 쉽고, 회고에서도 프론트와 백엔드, DB 데이터 기준 정합을 맞춘 경험이 공유됐습니다. 다만 전담 오너 부재로 인한 일관성 관리는 개선 여지가 있습니다.
- **Q3.** 팀장이 게이트웨이·인프라·CI/CD·staging 복구를 모두 맡으면 부하가 한 명에게 집중되는 위험은 없나요?
  - **A3.** 인프라와 CI/CD는 한 명이 표준을 잡되, 재사용 워크플로우와 멱등 자동화로 다른 멤버가 안전하게 재실행할 수 있게 설계했습니다. 그래서 운영 지식이 자동화 스크립트와 가드에 녹아 단일 인물 의존도를 낮췄습니다. 그래도 인프라 지식 분산은 팀 차원의 개선 포인트입니다.

## slide-11

**[대본]**

5주 스프린트는 5월 12일부터 6월 15일까지 22영업일로 진행했습니다. 사전 기획에서 주제와 요구사항, ADR 0001부터 0003, 문서 체계를 확정하고, 1주차에 K8s·DB·Kafka·ArgoCD 셋업과 4개 서비스 골격, Flutter 쉘을 세웠습니다. 2주차에 SM-2 복습과 AI 카드 골격, 청킹·BM25, Avro v1으로 핵심을 만들고, 3주차에 전 producer 발행과 게이미피케이션, RRF 하이브리드를 고도화하며 중간보고를 했습니다. 4주차에 notification·audit 소비와 모더레이션, dev·staging 배포를 검증하고, 5주차에 E2E 10시나리오와 P0 0건, SLA 실측, staging 가동으로 안정화했습니다. 발표 주간엔 코드를 동결하고 P0 핫픽스만 받습니다.

**예상 Q&A**

- **Q1.** 22영업일에 MSA·이벤트·검색·AI까지 다 넣는 건 일정이 빠듯한데, 스코프 관리는 어떻게 했나요?

- **Q2.** 이벤트 발행을 3주차, 소비를 4주차로 분리했는데 그 사이 통합 리스크는 어떻게 관리했나요?

- **Q1.** 6개 목표 중 핵심인 AI 카드와 검색 둘이 노란불인데, 제품의 핵심 가치가 미완이라는 뜻 아닌가요?
  - **A1.** 두 목표 모두 기능 골격과 동작은 구현됐고, 미완은 운영 검증과 정량 평가 부분이라 노란불로 솔직히 표시했습니다. G2는 운영 키와 스키마 정렬, G4는 골든셋 기반 정확도 측정이 남았습니다. 핵심 루프 G1은 E2E로 완주했고, 남은 건 운영 성숙도 차원의 명확한 개선 경로입니다.
- **Q2.** 계획에 없던 staging 실운영을 성취로 내세우는데, 계획 내 목표를 다 못 채운 상황에서 우선순위가 맞았나요?
  - **A2.** G1, G3, G5, G6 네 개를 달성하고 나머지 둘도 골격을 완성한 상태에서, GitOps 실배포와 SLA 실측이라는 현업 표준 운영 경험을 추가로 확보한 것입니다. 학습 목표가 현업 레퍼런스라 운영까지 완주한 가치가 큽니다. 다만 AI 체인 운영 검증을 더 앞당기지 못한 건 일정 배분의 아쉬움입니다.
- **Q3.** 16개 중 16개 Healthy와 24시간 소크로 운영 안정성을 주장하는데, 24시간이 프로덕션 검증으로 충분한가요?
  - **A3.** 24시간 소크와 16개 전 컴포넌트 Healthy, SLA 5개 실측은 staging 수준의 안정성 근거입니다. 본격 프로덕션 검증으로는 더 긴 기간과 부하 시나리오가 필요하다는 점은 인정합니다. 5주 일정 안에서 실운영에 근접한 검증까지 간 것이 의미이고, 장기 소크는 다음 단계입니다.

개선과 보완 과제입니다. 먼저 사용자 식별자 모델이 UUID와 Long 해시로 혼재돼 있어 전 서비스 UUID 정본으로 통일을 진행 중이고, P6 AI 카드 체인은 운영 키와 스키마 정렬이 미완이라 키 주입과 정본 정렬 후 SLA를 재측정할 계획입니다. 검색 정확도는 레이턴시만 실측했으니 골든셋을 구축해 MRR@10으로 정량 평가하고, Avro 버전 드리프트는 1.11.3과 1.12.0이 섞여 있어 정본 정렬과 CI 호환성 검증으로 잡습니다. Outbox 무한 재시도는 max-attempt와 DLQ 상한으로 막고, Streak 스텁과 첨부 미구현, 리더보드 DB 정렬 같은 잔여 기능도 마저 채울 계획입니다.

**예상 Q&A**

- **Q1.** UUID와 Long 해시 혼재를 설계 단계에서 못 잡은 건 큰 부채인데, 통일 마이그레이션 리스크는 어떻게 관리하나요?
  - **A1.** 식별자 모델 혼재는 회고에서도 설계 단계에서 확정 못 한 아쉬움으로 인정한 부분입니다. 전 서비스를 UUID 정본으로 통일하는 작업을 진행 중이며, 서비스 경계가 MSA로 나뉘어 있어 도메인별로 단계적 이행이 가능합니다. 이벤트 계약과 함께 정렬해 드리프트를 막는 방향입니다.
- **Q2.** Outbox 무한 재시도를 max-attempt와 DLQ 상한으로 막는다는데, 한도 초과 메시지는 어떻게 처리하나요?
  - **A2.** 현재 무한 재시도가 자원을 잠식할 수 있어 max-attempt를 두고 초과분은 DLQ로 보내되 DLQ에도 상한을 둘 계획입니다. DLQ로 격리된 메시지는 원인을 분석해 수동 또는 재처리 경로로 다룹니다. 재시도 폭주를 봉인하고 실패를 가시화하는 게 목표입니다.
- **Q3.** Avro 버전 드리프트가 있다는 건 이미 호환성 사고 위험이 현재진행형이란 뜻인데, 당장의 완화책은요?
  - **A3.** 1.11.3과 7.5.0, 1.12.0과 7.7.0이 섞인 드리프트를 정본 버전으로 정렬하는 게 핵심 대책입니다. 그 전까지는 BACKWARD 호환 정책과 schema-check CI가 깨지는 변경을 막는 완화 장치입니다. 정본 정렬과 CI 호환성 검증을 함께 강화해 드리프트를 근본적으로 닫겠습니다.

마지막으로 성과와 소감입니다. 잘한 점은 ADR을 남긴 정공법 아키텍처로 MSA·이벤트·Outbox·CQRS를 완주한 것, GitOps 실배포와 SLA 실측, 장애 복구, 관측성까지 운영을 완주한 것, 멀티레포와 크로스리뷰, 5종 문서로 7인이 5주를 완수한 협업 체계, 그리고 결함을 이슈에서 PR, 회귀 검증으로 닫는 사이클입니다. 아쉬운 점은 AI 체인 운영 검증을 발표 주까지 끌고 온 것, 식별자 모델 통일을 설계 단계에서 확정 못 한 것, 검색 품질 정량 평가가 미완인 것입니다. 그래도 각자 자기 트랙에서 현업 표준 도구체인을 실전 경험한 것이 경력으로 이어지는 성과라고 생각합니다.

**예상 Q&A**

- **Q1.** 아쉬운 점 세 가지가 모두 핵심 기능 완성도와 직결되는데, 다시 한다면 무엇을 가장 먼저 바꾸겠습니까?
  - **A1.** 식별자 모델 통일을 설계 단계에서 확정하는 것을 가장 먼저 바꾸겠습니다. 이 결정이 AI 체인 정렬과 이벤트 계약까지 파급되기 때문입니다. ADR로 의사결정을 남기는 방식은 유지하되, 식별자 같은 횡단 관심사를 더 일찍 못 박는 게 핵심 교훈입니다.
- **Q2.** 정공법 아키텍처를 강조하지만, 5주에 MSA·이벤트·CQRS를 다 넣은 게 학습용 과시 아니냐는 비판엔 어떻게 답하나요?
  - **A2.** 토이가 아니라 ADR로 트레이드오프를 기록하고, GitOps 실배포와 SLA 실측, 장애 복구까지 운영을 완주했다는 점이 과시와의 차이입니다. 결함도 이슈에서 PR, 회귀 검증으로 닫는 사이클로 관리했습니다. 학습 목표가 현업 표준 레퍼런스라, 표준 패턴을 끝까지 운영해 본 경험 자체가 성과입니다.
- **Q3.** 7인 5주 협업에서 가장 효과적이었던 체계 하나를 꼽는다면 무엇이고, 그 근거는요?
  - **A3.** SCOPE에서 HISTORY로 이어지는 5종 문서 체계와 멀티레포 크로스리뷰를 꼽습니다. 작업 기준을 코드보다 먼저 정해 7인의 방향을 정렬하고 재작업을 줄였기 때문입니다. 회고에서도 코드보다 작업 기준 문서를 먼저 두는 게 협업과 AI 활용에 유효했다고 공통적으로 정리됐습니다.
