Synapse 엔지니어링 · 신입(비전공자) 가이드

데이터는 어떻게 서비스 사이를 안전하게 흘러갈까?

PostgreSQL · CQRS · Kafka · Outbox · 복제본 · 순서 보장을 — 코드 한 줄 몰라도 — 비유로 이해하는 문서입니다. 우리 팀이 "왜 모든 걸 Kafka로 하지 않는지", 그리고 데이터를 어떻게 어긋나지 않게 맞추는지 끝까지 따라가 봅시다.

0한눈에 보는 큰 그림

Synapse는 하나의 거대한 프로그램이 아니라, 여러 개의 작은 서비스로 나뉘어 있습니다. 마치 회사 안의 여러 부서처럼요.

  • platform-svc — 로그인·결제·알림 (총무·경리부)
  • knowledge-svc — 노트·지식 그래프 (자료실)
  • learning-svc — 카드·복습·AI (교육부)
  • engagement-svc — 커뮤니티·게임요소(XP·뱃지) (동아리)

부서가 나뉘면 좋은 점이 많지만, 한 가지 숙제가 생깁니다. "한 부서에서 생긴 일을, 다른 부서에 어떻게 정확히 알려주지?" 예를 들어 카드를 복습하면(교육부 일), 동아리에서 XP를 올려줘야 합니다(동아리 일). 이 "알림과 데이터 맞추기"가 바로 이 문서의 주제입니다.

🏢 핵심 비유 한 줄

각 서비스 = 부서, 데이터베이스 = 부서의 캐비닛, Kafka = 부서 사이를 오가는 사내 우편 시스템. 이 문서는 "어떤 일은 내 책상에서 끝내고, 어떤 일만 우편으로 보내야 하는가"를 정리합니다.

1먼저, DB와 "CQRS"가 뭐죠?

데이터베이스(DB)는 데이터를 보관하는 거대한 표(엑셀 시트)들의 모음입니다. 우리는 PostgreSQL(줄여서 "포스트그레")를 씁니다.

CQRS는 어려운 영어 약자지만 뜻은 단순합니다: "데이터를 쓸 때 쓰는 표와, 읽을 때 쓰는 표를 나누자"는 아이디어입니다.

🍳 비유: 식당의 주방과 메뉴판

주방(쓰기 모델)은 재료를 다듬고 요리하는 복잡한 공간입니다. 손님이 들어와서 볼 곳이 아니죠. 메뉴판(읽기 모델)은 손님이 빠르게 보고 고를 수 있게 예쁘게 정리된 것입니다.

같은 음식 정보지만 용도가 달라서 모양이 다릅니다. CQRS는 "주방과 메뉴판을 따로 두자"는 것뿐입니다. 메뉴판(읽기용 표)을 따로 두면 손님 응대(조회)가 빨라집니다.

그러면 질문이 생깁니다. "주방에서 새 메뉴가 생기면, 메뉴판은 어떻게 따라서 바뀌지?" → 이게 바로 "읽기 모델 동기화(맞추기)" 문제이고, 여기서부터 사람들이 헷갈리기 시작합니다.

2"그럼 그 맞추기를 전부 Kafka로 하면 안 돼요?"

아주 좋은 질문이고, 이 문서가 생긴 이유입니다. 먼저 Kafka가 뭔지부터.

🚚 비유: Kafka = 우체국(택배 시스템)

Kafka는 한 서비스가 "이런 일이 생겼어요!"라고 편지(이벤트)를 부치면, 그걸 관심 있는 다른 서비스들에게 배달해 주는 우체국입니다. 보내는 쪽과 받는 쪽이 서로 직접 만나지 않아도 됩니다(느슨한 결합).

그런데 모든 데이터 맞추기를 우체국(Kafka)으로만 하면 이상해집니다. 생각해 보세요:

😵 모든 걸 우편으로 하면…

내 책상 위 서류 두 장을 맞추려고 나 자신에게 우편을 부친다고 상상해 보세요. 봉투에 넣고, 우표 붙이고, 우체통에 넣고, 며칠 뒤 내가 받습니다. 바로 옆 서랍에 넣으면 1초인데 말이죠.

그래서 직관이 맞습니다. Kafka는 "부서 사이"를 넘어갈 때만 쓰는 것이고, "한 부서 안"에서 표를 맞추는 일에는 쓰지 않습니다. 이게 복잡함의 80%를 없애는 한 문장입니다.

⭐ 외워둘 한 문장

Kafka는 서비스 사이를 잇는 우편이다. 서비스 안의 표 맞추기에는 쓰지 마라 — 그건 DB와 같은 트랜잭션으로 끝낸다.

💡 헷갈림의 진짜 원인

사람들은 "CQRS = Kafka"라고 묶어서 생각합니다. 사실은 완전히 별개입니다. CQRS(쓰기/읽기 표 나누기)는 DB 하나 안에서 Kafka 없이도 됩니다. Kafka는 그저 부서를 넘나드는 운송수단일 뿐이에요.

3동기화 4단계 사다리 — 싼 것부터 골라라

"표를 맞추는 방법"은 한 가지가 아닙니다. 비용이 다른 여러 방법이 있고, 가장 싸고 단순한 것부터 고르면 됩니다. Kafka(L3)는 사다리 맨 위, 정말 필요할 때만.

단계방법 (비유)언제 쓰나Kafka?
L0그냥 VIEW
계산해서 바로 보여주기 (즉석 조합)		읽기 모양이 단순한 조합일 때. 맞출 것도 없음불필요
L1같은 트랜잭션에 같이 쓰기
서류 쓰면서 옆 장부도 같이 기입		항상 정확히 맞아야 하고 같은 서비스일 때 (예: 누적 XP)불필요
L2Modulith 사내 메모
같은 건물 안 다른 부서에 쪽지		같은 서비스 안 다른 모듈에 비동기로 알릴 때 (우편 불필요)불필요

"이 표가 쓰기와 같은 서비스에 있나?" → YES면 L0~L2로 내려가세요(Kafka 금지). NO(다른 서비스/딴 저장소)면 그제서야 L3(Kafka)입니다.

실제로 Synapse에서 Kafka가 꼭 필요한 곳은 의외로 적습니다: 노트→검색엔진 색인, 카드복습→동아리 XP, 여러 이벤트→알림함처럼 부서를 넘거나 다른 저장소로 갈 때뿐입니다. 노트의 백링크나 리더보드 집계는 이미 각자 부서 안(트랜잭션·정기배치)에서 처리합니다.

4L3에서 생기는 함정: "dual-write 문제"

부서를 넘는 일(L3)에서 순진하게 하면 이렇게 됩니다: "① DB에 저장하고, ② 곧바로 Kafka로 편지 부치기." 코드 두 줄이죠. 그런데 여기에 위험한 틈이 있습니다.

⚠️ 두 가지 사고 시나리오

사고 A: DB 저장은 성공했는데, 그 직후 Kafka 우체국이 잠깐 고장 → 편지를 못 부침. 동아리는 XP를 영영 모릅니다. (이벤트 유실)

사고 B: 편지는 먼저 부쳤는데, DB 저장이 막판에 실패(롤백) → 일어나지도 않은 일의 편지가 돌아다닙니다. (유령 이벤트)

문제의 핵심: "DB 저장"과 "Kafka 발송"은 하나로 묶을 수 없습니다. 둘 중 하나만 성공하는 순간이 반드시 존재하죠. 이걸 dual-write(이중 쓰기) 문제라고 부릅니다.

해결책은 발상의 전환입니다 👇

5해결책: Outbox 패턴 = "보내는 편지함"

📮 비유: 집 앞 편지함에 먼저 둔다

편지를 직접 우체통까지 들고 가지 않습니다. 대신 내 책상 서랍(=DB) 안의 "보내는 편지함"에 편지를 같이 넣어둡니다. 서류 작업과 편지함 넣기는 같은 한 번의 동작이라 둘 다 되거나 둘 다 안 됩니다(사고 A·B 소멸).

그 다음, 집배원(Relay)이 주기적으로 편지함을 열어보고, 있는 편지를 진짜 우체국(Kafka)으로 가져가 부칩니다.

여기서 "보내는 편지함"이 바로 DB 안의 outbox_event라는 표입니다. 흐름으로 보면:

① 도메인 서비스비즈니스 표 수정 + 편지함(outbox)에 기록
둘이 한 트랜잭션 (Kafka 호출 ❌)
→
② 편지함 (outbox_event)DB 안의 표.
"부칠 편지"가 PENDING 상태로 쌓임
→
③ 집배원 (Relay)주기적으로 편지함 확인 →
Kafka로 발송 → PUBLISHED로 표시
→
④ Kafka → 다른 서비스받는 서비스가 편지를 읽고
자기 일을 함

이렇게 하면 ①에서 "DB냐 Kafka냐" 갈림길이 사라집니다. ①은 DB 하나만 건드리니까 안전하고, 발송은 ③ 집배원이 책임지고 성공할 때까지 재시도합니다.

// ① 도메인 서비스: 카드 복습 완료 — 한 트랜잭션 안에서
@Transactional
public void completeReview(ReviewCommand cmd) {
    Card card = cardRepo.findById(cmd.cardId());
    card.review(cmd.rating());        // SM-2 복습 로직(비즈니스)
    cardRepo.save(card);              // (가) 비즈니스 표 수정

    outboxRecorder.record(            // (나) 같은 트랜잭션에 편지 적재
        "card", card.getId(), "card.reviewed", payloadOf(card));
}   // 커밋 → (가)와 (나)가 함께 성공/실패. Kafka는 아직 모름 → 안전
💡 같은 건물(서비스) 안이라면?

편지함조차 필요 없습니다. 우리 프로젝트는 Spring Modulith라는 "사내 메모 시스템"(L2)이 있어서, 같은 서비스 안 모듈끼리는 우편(Kafka) 없이 메모로 알립니다. Outbox는 건물 밖으로 나가는 편지에만 씁니다.

6편지함을 열어보자: outbox_event 표

편지함은 그냥 DB 표 하나입니다. 각 칸(열)이 무슨 뜻인지 봅시다 — 어렵지 않아요.

칸 이름쉬운 설명
id편지마다 붙는 고유 번호
aggregate_type / aggregate_id"무엇에 대한" 편지인가 (예: card / 특정 카드 ID) — 순서를 지킬 단위
aggregate_version그 대상의 "몇 번째 변경"인지 — 나중에 순서 꼬임을 막는 도장 번호
event_type편지 종류 (예: card.reviewed) → 어느 우편함(토픽)으로 갈지 결정
event_id편지 고유 번호. 받는 쪽이 "이거 이미 처리했나?" 확인용(중복 방지)
payload편지 내용물(JSON). 사람이 읽을 수 있게 저장 → 문제 생기면 바로 확인
status편지 상태: PENDING(대기) → PUBLISHED(발송완료) / FAILED / DEAD(포기)
retry_count / next_attempt_at실패 시 "몇 번 재시도했나 / 다음엔 언제 다시 시도할까"
last_error마지막 실패 이유 메모
 
-- 편지함 표 (요약)
CREATE TABLE outbox_event (
    id                UUID PRIMARY KEY,
    aggregate_type    VARCHAR(64)  NOT NULL,   -- "card", "note", "user"
    aggregate_id      UUID         NOT NULL,   -- 순서 보장 단위
    aggregate_version BIGINT       NOT NULL,   -- 도장 번호(순서/중복 방어)
    event_type        VARCHAR(128) NOT NULL,   -- 편지 종류 → 토픽
    event_id          UUID         NOT NULL UNIQUE,
    payload           JSONB        NOT NULL,   -- 내용물(읽을 수 있게)
    status            VARCHAR(16)  NOT NULL DEFAULT 'PENDING',
    retry_count       INT          NOT NULL DEFAULT 0,
    next_attempt_at   TIMESTAMPTZ  NOT NULL DEFAULT now(),
    created_at        TIMESTAMPTZ  NOT NULL DEFAULT now(),
    published_at      TIMESTAMPTZ,
    last_error        TEXT
);
-- 집배원이 "대기중 편지만" 빨리 찾도록 하는 색인
CREATE INDEX idx_outbox_due ON outbox_event (next_attempt_at, created_at)
    WHERE status = 'PENDING';

이 표는 우리 팀의 02_ERD 문서 §2.3.A에 정식으로 등록해 두었습니다(이번에 추가됨).

7그런데 이 표는 누가 만들죠? — Flyway 마이그레이션

6장에서 보여드린 CREATE TABLE outbox_event ...는 글로만 있는 게 아닙니다. 실제 데이터베이스에 만들어 두어야 합니다. 그 일을 Flyway라는 도구가 합니다.

📓 비유: DB 변경을 적는 "일기장"

Flyway는 DB에 가한 모든 변경을 시간순 일기장처럼 모아 둡니다. 한 장씩 페이지(파일)에 적고, 페이지엔 일련번호(V1, V2, V3…)가 붙어 있죠. 서비스가 시작될 때 Flyway가 "어디까지 적용됐나"를 보고, 새 페이지만 차례로 DB에 실행합니다.

한 번 적은 일기는 고치거나 지우지 않습니다. 새 변경이 필요하면 새 페이지를 추가합니다(예: 컬럼 추가는 V4 새 파일로). 일기장의 무결성은 체크섬으로 검사돼요 — 옛 페이지를 수정하면 다음 시작 때 빌드가 막힙니다.

파일 이름 규칙: V{번호}__{설명}.sql

src/main/resources/db/migration/
├── V1__init_extensions.sql            // V + 번호 + __ (밑줄 두 개) + 설명
├── V2__create_users.sql
├── V3__create_outbox_event.sql        // 6장의 outbox 표는 여기서 만들어진다
└── V4__add_outbox_aggregate_version.sql   // 컬럼 추가는 새 V번호로

각 서비스(platform-svc, learning-card, …)가 자기 마이그레이션 폴더를 갖습니다. 예를 들어 platform-svc는 V1~V28, learning-card는 V1~V9까지 있습니다(서비스마다 자기 스키마만 책임 — ADR-0003).

outbox_event 마이그레이션 — 실제 한 페이지

-- V{n}__create_outbox_event.sql  (6장의 DDL을 그대로 일기로 적은 모습)
CREATE TABLE outbox_event (
    id                UUID PRIMARY KEY,
    tenant_id         UUID NOT NULL,
    aggregate_type    VARCHAR(64)  NOT NULL,
    aggregate_id      UUID NOT NULL,
    aggregate_version BIGINT NOT NULL DEFAULT 0,
    event_type        VARCHAR(128) NOT NULL,
    event_id          UUID NOT NULL,
    payload           JSONB NOT NULL,
    headers           JSONB,
    status            VARCHAR(16)  NOT NULL DEFAULT 'PENDING',
    retry_count       INT NOT NULL DEFAULT 0,
    next_attempt_at   TIMESTAMPTZ  NOT NULL DEFAULT now(),
    created_at        TIMESTAMPTZ  NOT NULL DEFAULT now(),
    published_at      TIMESTAMPTZ,
    last_error        TEXT,
    CONSTRAINT uq_outbox_event_id UNIQUE (event_id)
);

CREATE INDEX idx_outbox_due ON outbox_event (next_attempt_at, created_at)
    WHERE status = 'PENDING';

서비스가 켜질 때 Flyway가 하는 일

  1. 1일기장(DB) 확인 — flyway_schema_history 라는 특수 표를 보고 "어디까지 적용됐나"를 확인합니다.
  2. 2새 페이지 찾기 — 미적용된 V번호를 작은 것부터 정렬합니다.
  3. 3차례로 실행 + 기록 — 각 SQL 실행 → 성공하면 일기장에 도장(체크섬·실행 시각)을 찍습니다.
  4. 4분산 락 — Kubernetes에서 같은 서비스 파드가 여러 개 동시에 켜져도 딱 한 명만 실행합니다(Flyway 내장 락).
⚠️ 신입이 가장 자주 하는 실수 두 가지

① 이미 적용된 V번호 파일을 수정하기. "오타 하나만…" 하다가 다른 환경(스테이징·운영)과 체크섬이 안 맞아 서비스 시작이 막힙니다. 변경이 필요하면 반드시 새 V번호로 추가하세요(예: V3의 컬럼 정정은 V4에서 ALTER TABLE로).

② 운영 DB에 직접 ALTER TABLE 실행. 일기장에 안 적힌 변경은 다른 환경엔 없으니까요. 모든 스키마 변경은 새 V번호 마이그레이션 파일로만.

🧭 이 가이드 전체와의 관계

이 가이드의 모든 표 — outbox_event(편지함) · user_ref(복제본) · processed_events(처리 도장) · flyway_schema_history(일기장 자체) — 는 전부 Flyway 마이그레이션으로 만들어집니다. 가이드의 나머지 흐름은 "표가 이미 있다"고 가정하고 설명했지만, 실제로는 마이그레이션이 가장 먼저 실행되어 표가 생긴 뒤에야 코드가 동작합니다.

정식 표준: 02_ERD §2.6 마이그레이션 전략(네이밍·되돌리기·CONCURRENTLY·RLS 분리) · 18 §4.1.5 Flyway 10.x(설정 가이드·트러블슈팅).

8집배원: "Polling Relay"

Polling(폴링)은 "주기적으로 확인하기"란 뜻입니다. Relay(릴레이)는 "옮겨 전달하기". 합치면 "편지함을 주기적으로 확인해서 편지를 우체국으로 옮기는 일꾼"입니다.

  1. 1편지함 확인 — 1초마다 "대기(PENDING)이고 지금 보낼 때가 된" 편지를 한 묶음 꺼냅니다.
  2. 2발송 — 각 편지를 Kafka로 보내고, 우체국이 "잘 받았다"고 확인해주면 → PUBLISHED로 표시.
  3. 3실패하면 — 횟수를 올리고, 점점 더 긴 간격으로 다시 시도(지수 백오프). 너무 여러 번 실패하면 DEAD로 옮기고 사람에게 알림.
  4. 4청소 — 발송 끝난 오래된 편지는 정기적으로 삭제(편지함 비우기).
// 집배원: 1초마다 편지함을 비운다
@Scheduled(fixedDelay = 1000)
@SchedulerLock(name = "outbox-relay")   // ★ 집배원이 동시에 여럿이면 안 되게 잠금(ShedLock)
@Transactional
public void relay() {
    for (OutboxEvent e : repo.findDueBatchForUpdate(200)) {
        try {
            OutboxMessage m = factory.create(e);   // 표준 택배로 포장(9장 참고)
            kafka.send(m.topic(), m.key(), m.value()).get();  // 발송 + 확인
            e.markPublished();                     // 성공
        } catch (Exception ex) {
            e.markFailed(ex.getMessage());         // 실패 → 재시도 예약 or DEAD
        }
    }
}
🔒 "FOR UPDATE SKIP LOCKED" — 집배원끼리 안 부딪히기

편지함에서 편지를 꺼낼 때 이 마법 주문을 씁니다. 뜻: "내가 집은 편지엔 다른 집배원이 손대지 마. 그리고 남이 이미 집은 건 건너뛰고(SKIP) 다음 걸 집어." 덕분에 여러 집배원이 같은 편지를 두 번 부치는 사고가 안 납니다.

-- 대기 편지를 꺼내는 SQL (집배원 충돌 방지)
SELECT * FROM outbox_event
WHERE status = 'PENDING' AND next_attempt_at <= now()
ORDER BY created_at
LIMIT 200
FOR UPDATE SKIP LOCKED;   -- 남이 집은 건 건너뛴다
📬 "최소 한 번은 배달된다"(at-least-once)

집배원이 편지를 부친 직후, "발송완료" 표시를 하기 전에 쓰러질 수 있습니다. 그러면 깨어나서 같은 편지를 또 부칩니다. 그래서 받는 쪽은 "같은 편지 두 번 받아도 괜찮게" 만들어야 합니다(중복 방지 = 멱등). 10장에서 다룹니다.

9OutboxMessageFactory: 편지를 "표준 택배"로 포장

편지함의 내용물(payload)은 우리가 읽기 쉬운 JSON입니다. 하지만 다른 서비스에 보낼 땐 회사 표준 규격으로 포장해야 합니다. 그 포장 담당이 OutboxMessageFactory입니다. 두 가지를 합니다:

내용물 → 표준 상자JSON 내용물을 Avro라는 표준 규격 데이터로 변환
(= 모든 부서가 똑같이 해석)
+
송장 붙이기CloudEvents 송장:
보낸곳·받을종류·운송장번호·시각·테넌트
📦 비유: Avro = 규격 상자, CloudEvents = 택배 송장

Avro는 "모두가 똑같은 모양으로 접는 상자"입니다. 보내는 쪽과 받는 쪽이 상자 모양(데이터 구조)을 미리 합의해 두면, 받자마자 정확히 풀 수 있습니다. 합의 장부는 Schema Registry가 관리합니다.

CloudEvents는 상자 겉에 붙이는 표준 송장입니다: 누가 보냈고(source), 무슨 종류이고(type), 운송장 번호(id), 언제(time), 어느 회사(tenant) 것인지. 우리 회사 송장 양식은 CloudEventEnvelope.avsc에 정의돼 있습니다.

송장 정보는 택배 상자 겉(Kafka 헤더)에 붙고, 내용물 상자는 안(Kafka 값)에 들어갑니다. 이 방식을 "CloudEvents 바이너리 모드"라고 부릅니다.

🧱 표준: 처음부터 Avro로 (JSON으로 시작하지 않기)

"일단 JSON으로 보내다가 나중에 Avro로 바꾸자"는 흔한 함정입니다. 도중에 규격을 바꾸면 받는 쪽이 못 알아들어 운영 사고로 이어지죠. 그래서 Synapse는 처음부터 Avro + Schema Registry를 표준으로 씁니다(기술스택 문서 18 §5.5에 "JSON→Avro 전환은 안티패턴"으로 못 박혀 있어요).

헷갈리지 마세요: 편지함(outbox)의 payload는 사람이 읽기 쉽게 JSON으로 보관하지만, 우편으로 실제 나갈 때는 항상 Avro 상자로 포장됩니다 — 그 포장을 이 9장의 Factory가 합니다.

// 포장 담당. 편지(outbox 행) → 표준 택배(OutboxMessage)
public OutboxMessage create(OutboxEvent e) {
    EventTypeDescriptor d = registry.require(e.getEventType()); // 편지종류별 규칙 찾기
    JsonNode node       = om.readTree(e.getPayload());          // 내용물 JSON 읽기
    SpecificRecord avro = d.mapper().apply(node);   // ★ JSON → Avro 규격 상자로 변환
    String key          = e.getTenantId() + ":" + e.getAggregateId(); // 묶음 키(순서용)
    return new OutboxMessage(d.topic(), key, avro, ceHeaders(e, d)); // + 송장 헤더
}

// 송장(CloudEvents) 만들기 — 상자 겉면 라벨들
private List<Header> ceHeaders(OutboxEvent e, EventTypeDescriptor d) {
    return List.of(
        header("ce_id",       e.getEventId()),   // 운송장 번호
        header("ce_source",   d.source()),       // 보낸 부서 "/learning-svc/srs"
        header("ce_type",     d.ceType()),       // "com.synapse.learning.ReviewCompleted"
        header("ce_time",     e.getCreatedAt()),
        header("ce_tenantid", e.getTenantId()),  // 어느 회사(테넌트)
        header("content-type", "application/avro"));
}

편지 종류별 "포장 규칙"은 한 곳에 등록

편지 종류(card.reviewed 등)마다 "어느 토픽으로, 어떤 상자 모양으로" 보낼지 규칙을 레지스트리에 모아둡니다. 새 편지 종류가 생기면 여기 한 줄만 추가하면 됩니다.

// JSON의 각 칸을 Avro 상자의 각 칸에 1:1로 옮기는 "매퍼" (타입 안전)
static SpecificRecord reviewCompleted(JsonNode n) {
    return ReviewCompleted.newBuilder()
        .setCardId(n.get("cardId").asText())
        .setUserId(n.get("userId").asText())
        .setRating(Rating.valueOf(n.get("rating").asText())) // AGAIN|HARD|GOOD|EASY
        .setNextReviewAt(n.get("nextReviewAt").asText())
        .build();
}
🧩 왜 한 줄씩 직접 매핑해요? 자동으로 하면 안 되나?

JSON을 Avro 상자로 "자동" 변환하면 특수한 경우(빈 값, 선택 항목, enum)에서 조용히 깨지기 쉽습니다. 종류마다 매퍼를 직접 쓰면 약간 귀찮지만, 상자 모양이 바뀌면 컴파일 단계에서 바로 빨간 줄이 떠서 사고를 미리 막습니다. 안전이 귀찮음을 이깁니다.

10핵심 질문: 서비스마다 "A 표"를 따로 들고 있을 때, 어떻게 맞추죠?

질문을 다시 풀면: A 서비스가 진짜 주인인 데이터(예: 사용자 정보)를, B 서비스도 자기 캐비닛에 사본으로 들고 있습니다. A에서 바뀌면 B의 사본은 어떻게 따라 바뀌어야 할까요?

원칙 0 — 주인은 딱 하나 (Source of Truth)

📚 비유: 도서관 본관과 분관

본관(A 서비스)만 장서 목록을 고칠 수 있습니다. 분관(B 서비스)은 본관 목록의 읽기 전용 사본을 둘 뿐, 절대 자기 마음대로 고치지 않습니다. 분관이 멋대로 고치기 시작하면 "어느 게 진짜냐" 싸움이 납니다.

원칙 1 — 그런데 B가 정말 사본을 가져야 할까?

사본은 공짜가 아닙니다(맞추는 수고 + 약간의 시차). 먼저 따져보세요:

B의 사용 방식권장사본?
A 데이터로 자주 목록·정렬·검색 (예: 리더보드에 이름 표시)이벤트로 사본 맞추기 (아래)✅
가끔 한 건만 필요그때 A에 직접 물어보기(내부 API)❌
화면에 보여줄 때만 합치면 됨Gateway에서 조합❌
거의 안 바뀌는 참조 데이터잠깐 캐시△

원칙 2 — 사본을 둔다면: "상태를 통째로 실어 보내기"

A는 Outbox로 바뀐 내용 전체 + 도장 번호(version)를 편지에 담아 보냅니다. B는 A에게 되묻지 않고, 편지 내용만으로 자기 사본을 갱신합니다.

🚫 하면 안 되는 것

"편지엔 ID만 담고, 자세한 건 B가 A에 다시 전화해서 물어보기" → A에 전화가 폭주하고, B는 A가 아프면 같이 멈춥니다. 사본을 둔 의미가 사라지죠. 편지에 필요한 내용을 다 담으세요.

// B(동아리 서비스)가 A(사용자 서비스)의 변경 편지를 받아 사본 갱신
@KafkaListener(topics = {"user.registered", "user.updated"})
public void onUserChanged(UserChanged ev, @Header("ce_id") UUID eventId) {
    if (processedEvents.exists(eventId)) return;   // ★ 이미 처리한 편지면 무시(중복 방지)

    // 있으면 갱신, 없으면 추가(UPSERT). 단, "더 최신 도장"일 때만 덮어쓴다.
    upsert("""
      INSERT INTO user_ref (user_id, display_name, version, synced_at)
      VALUES (:id, :name, :ver, now())
      ON CONFLICT (user_id) DO UPDATE SET
          display_name = EXCLUDED.display_name,
          version      = EXCLUDED.version
      WHERE EXCLUDED.version > user_ref.version
    """, ev);                                  // ← 도장 번호 비교가 핵심(11장)
    processedEvents.save(eventId);
}

삭제도 마찬가지입니다: user.deleted 편지가 오면 사본에 "삭제됨" 표시를 합니다(이걸 tombstone=묘비라고 부릅니다). 그리고 사본이 망가지면 편지를 처음부터 다시 받아 통째로 재생성할 수 있어야 합니다.

이 사본 표(user_ref)도 02_ERD §2.3.A에 정식 등록했습니다.

11편지가 순서 뒤바뀌어 도착하면? — 순서 보장

집배원을 여러 명 두면 빨라집니다. 그런데 위험이 생깁니다: 같은 대상에 대한 편지 두 통("이름을 A로 바꿈" → "이름을 B로 바꿈")을 서로 다른 집배원이 들고 가다가, 나중 편지가 먼저 도착할 수 있습니다. 그러면 사본 이름이 옛날 값(A)으로 남아버립니다.

🗂️ 표준: 편지를 어느 칸에 넣을지 — "테넌트ID : 대상ID"

우체국(Kafka)에는 여러 칸(파티션)이 있고, 같은 칸에 들어간 편지는 넣은 순서가 그대로 유지됩니다. Synapse는 편지를 넣을 칸을 정하는 열쇠(파티션 키)를 테넌트ID : 대상ID로 통일했습니다. 그래서 같은 대상(예: 같은 사용자)에 대한 편지는 항상 같은 칸으로 가서 순서가 지켜집니다.

테넌트ID만으로 칸을 정하면(옛 방식) 한 회사의 모든 편지가 한 칸에 몰려 느려집니다. 그래서 대상ID까지 붙여 칸을 잘게 나눕니다. 이 표준은 기술스택 문서 18 §5.4·03-A §A.11.3에 동일하게 적혀 있어요.

구역 나누기집배원 N명이 "대상 ID 끝번호"로 구역을 나눠 맡음 (한 대상은 늘 같은 사람)유지빠름물량 늘면

⭐ 가장 중요한 철칙

집배원의 순서 맞추기는 "있으면 좋은 것"이고, 진짜 정합성은 "받는 쪽 도장 비교"에서 옵니다.

편지에 도장 번호(version)를 찍어 보내고, 받는 쪽은 "내 사본 도장보다 큰 번호일 때만 덮어쓴다"(WHERE EXCLUDED.version > user_ref.version)고 하면 — 편지가 뒤죽박죽 도착해도 결과는 항상 최신으로 수렴합니다. 10장의 그 한 줄이 바로 이것입니다.

🔢 비유: 택배 송장의 "버전 도장"

편지마다 "이건 3번째 변경", "4번째 변경" 도장이 찍혀 있습니다. 받는 사람은 지금 내가 가진 게 3번인데, 방금 2번이 늦게 도착했네? → 무시. "4번이 왔네? → 반영." 이렇게만 하면 순서가 꼬여도 절대 거꾸로 가지 않습니다.

덤: Kafka 발송 설정 한 가지

집배원이 한 명이라도, 발송 재시도 중에 순서가 꼬일 수 있어서 이 설정을 꼭 켭니다:

spring.kafka.producer:
  acks: all
  properties:
    enable.idempotence: true   # 재시도해도 순서·중복 보장

DB 저장과 Kafka 발송을 따로 하면 한쪽만 성공할 수 있는 위험.
Transactional Outbox
이벤트를 DB의 "보내는 편지함" 표에 같이 저장해 dual-write를 없애는 방법.
Polling Relay
편지함을 주기적으로 확인해 Kafka로 옮기는 일꾼(집배원).
멱등(Idempotent)
같은 편지를 두 번 처리해도 결과가 같게 만드는 것. (중복 방어)
Avro
모두가 똑같이 해석하는 표준 데이터 규격(규격 상자).
Schema Registry
그 상자 모양(스키마)을 합의·관리하는 장부.
CloudEvents
이벤트에 붙이는 표준 송장(누가·무엇을·언제·운송장번호).
복제본(Replica)
다른 서비스가 주인인 데이터를 읽기 전용으로 복사해 둔 표.
Source of Truth (SoT)
데이터의 "진짜 주인". 고칠 수 있는 곳은 여기 하나뿐.
version 가드
"더 최신 도장일 때만 덮어쓰기" 규칙. 순서 꼬임 방어.
tombstone
"삭제됨" 표시(묘비). 복제본에서 삭제를 나타냄.
at-least-once
"최소 한 번은 배달"(=가끔 두 번). 그래서 멱등이 필요.
ShedLock
여러 서버 중 한 명만 정해진 작업을 하게 잠그는 도구.
Spring Modulith
한 서비스 안 모듈끼리 우편 없이 알리는 "사내 메모" 구조.
Flyway
DB 변경을 시간순 일기장처럼 V{번호}__{설명}.sql 파일로 모아 두는 마이그레이션 도구. 서비스 시작 시 자동 적용.
마이그레이션
DB 표·인덱스·컬럼 변경을 코드 옆에 SQL 파일로 두고 버전 관리하는 것. 한 번 적용된 파일은 수정 ❌, 새 V번호로 추가만.
flyway_schema_history
Flyway가 "어디까지 적용했나"를 기록하는 특수 표(=일기장 자체). 체크섬으로 무결성 검사.

🎯 한 장 요약 (이것만 기억하면 됨)

1 큰 원칙

Kafka는 서비스 사이를 잇는 우편. 서비스 안의 표 맞추기는 DB 트랜잭션이나 Modulith로 끝낸다. 새 읽기표마다 "같은 서비스인가?"를 먼저 물어라.

2 부서를 넘을 때(L3)는 항상 Outbox

DB 저장과 같은 트랜잭션으로 편지함(outbox_event)에 적재 → 집배원(Relay)이 Kafka로 발송. dual-write 사고가 사라진다.

3 받는 쪽은 두 가지를 항상

① 멱등(같은 편지 두 번 와도 OK, event_id로 확인) ② version 가드(더 최신 도장일 때만 덮어쓰기). 이 둘이면 중복·순서 꼬임이 모두 막힌다.

4 복제본 규칙

주인은 하나(SoT). 복제본은 읽기 전용. 필요한 필드만 복제. 이벤트엔 상태를 통째로 담는다. 망가지면 리플레이로 재생성.
