01

프로젝트 개요

주제 · 선정 배경 · 기획 의도

먼저 프로젝트 개요입니다 — 무엇을, 왜 만들었는지부터.

주제 · 선정 배경

SYNAPSE — PKM(개인 지식 관리)과 SRS(간격 반복 학습)를 하나의 워크플로우로 통합하고, 그 사이를 AI(LLM)가 자동 연결하는 멀티테넌트 학습 SaaS. 포지셔닝: "Obsidian + Anki + RAG 융합"
#기존 도구의 문제원인
1워크플로우 단절PKM과 SRS 분리 — 노트를 카드로 수동 복사
2맥락 상실복습 카드가 원본 노트와 연결되지 않음
3카드 제작 노동손으로 만드는 카드가 학습 의욕 저하
4관계 시각화 부재노트 간 연결을 한눈에 볼 수 없음
5검색 한계키워드 매칭만 가능, 의미 기반 검색 없음
기존 학습 도구는 PKM과 SRS 두 진영으로 갈라져 있습니다. 옵시디언·노션은 노트는 좋지만 과학적 복습이 없고, 앙키는 복습은 강력하나 카드를 손으로 만들고 원본 맥락과 끊깁니다. 표의 다섯 문제 — 워크플로우 단절·맥락 상실·카드 제작 노동·시각화 부재·검색 한계 — 가 저희가 푼 문제이고, 목표는 G1 통합부터 G6 수익화까지 여섯 가지입니다.

차별화 포인트

제품 — "통합과 자동화"

  • PKM+SRS 단일 플랫폼 — 카드가 원본 노트에 링크, 맥락 유지
  • AI 카드 자동 생성 — LLM이 노트 분석, 제작 노동 제거
  • 위키링크 지식 그래프 — 백링크·PageRank·2D 시각화
  • 하이브리드 검색 — BM25 × 임베딩을 RRF로 융합

토이 프로젝트가 아닌 의사결정 기록(ADR)을 남기는 아키텍처 학습

차별화는 제품과 아키텍처 두 축입니다. 제품은 통합과 자동화 — 카드가 항상 원본 노트에 링크되고, LLM이 노트를 분석해 카드를 자동 생성하며, 위키링크로 지식 그래프를, BM25와 임베딩을 RRF로 융합해 검색합니다. 아키텍처는 토이가 아니라 ADR 세 건으로 의사결정을 기록한 현업형 MSA 정공법이라는 점이 핵심입니다.

04

프로젝트 수행 경과

한 사용자의 학습 여정으로 보는 수행 경과 — 진입 → 노트 → AI 카드 → 복습 → 보상 → 검색 → 백본 → 시연 영상

핵심인 수행 경과입니다. 담당자 나열이 아니라 한 사용자의 학습 여정 흐름으로 보여드립니다.

사용자 시나리오 한 바퀴 — 노트가 도는 학습 루프

단계사용자 동작 · 핵심이벤트 / 기술담당
① 진입·계정가입·로그인·결제·알림user-registered 발행김해준
② 노트·그래프마크다운·위키링크·백링크note-created · PageRank김현지
③ AI 카드노트 → 카드 자동 생성소비 → Claude · RAG(SSE)김나경
④ 복습(SRS)SM-2 4버튼·세션review-completed 발행조유지
⑤ 보상·커뮤니티XP·레벨·배지·공유소비 → level-up · 알림 환류한승완
⑥ 검색·재발견의미 검색·AI 튜터임베딩 · BM25 × RRF박은서
⑦ 백본Gateway·인프라·운영Kafka(Outbox) · EKS · SLA김민구

노트 한 장이 카드 → 복습 → 보상 → 검색으로 순환 — 아래 ①~⑦이 그 순서, ⑦ 백본 위에서 전 구간이 돈다.

여기서부터 04입니다. 한 사용자가 제품을 한 바퀴 도는 흐름으로 봅니다 — 가입·로그인으로 들어와 노트가 카드가 되고, 복습이 보상과 알림으로 돌아오고, 검색으로 다시 발견되는 순환입니다. 표의 ①부터 ⑦까지가 그 순서이고, 전 구간이 게이트웨이·카프카·인프라 위에서 돕니다. 이제 각 단계를 담당자별로 짚겠습니다.

① 진입·계정 — platform · 김해준 (트랙 A)

사용자가 들어오는 관문 — 인증·결제·알림·감사의 기반

  • 인증·테넌트 — OAuth(Google/GitHub/Apple) + JWT RS256 + MFA(TOTP), tenant self-service·초대·DB 기반 role
  • 결제(Stripe) — Free/Pro/Team 구독, 결제 이력·영수증·사용량 조회
  • 알림 — 인박스 + FCM/SES 발송(notification-send 소비), 미읽음 배지·클릭 라우팅
  • 감사·관리자 — 전 이벤트 소비 audit log(90일 보존), GDPR 데이터 요청, 시스템 설정·분석 API
  • 이벤트 — user-registered 발행(Transactional Outbox), 그룹 platform-svc-group
첫 단계, 진입입니다. 김해준 님의 플랫폼은 사용자가 들어오는 관문입니다. OAuth·JWT·MFA로 인증하고, 스트라이프로 결제하며, FCM·SES로 알림을 보내고, 모든 이벤트를 감사 로그로 90일 보존합니다. user-registered 이벤트를 Outbox로 발행해 — 뒤의 게이미피케이션이 이걸 받아 프로필을 만듭니다. 모든 서비스가 기대는 기반입니다.

② 노트가 지식이 된다 — knowledge · 김현지 (트랙 C-1)

쓴 노트가 링크로 엮여 지식 그래프가 된다

  • Markdown 노트 CRUD · 버전 이력·복원 · 태그 (소유자 격리 validateOwner)
  • 위키링크 파싱 · 백링크 · PageRank 그래프 (GraphQueryPort 포트 패턴)
  • Kafka→ES 자동 동기화 (Outbox AFTER_COMMIT · 멱등 Redis TTL 7d · DLQ)
두 번째, 노트가 지식이 되는 단계입니다. 김현지 님의 knowledge 노트 모듈입니다. 마크다운으로 쓰고 위키링크로 노트끼리 연결하면 백링크와 PageRank로 지식 그래프가 만들어집니다. 저장 즉시 Outbox AFTER_COMMIT 이벤트로 Elasticsearch에 동기화되고, 멱등 처리와 DLQ로 신뢰성을, 모든 메서드에 소유자 격리를 확보했습니다.

③ 노트가 카드가 된다 — AI 카드 생성 · learning-ai · 김나경 (트랙 D-2)

노트를 던지면 AI가 플래시카드를 만든다 — 제품의 핵심

  • FastAPI 골격 + LLM 이중화(Tenacity → OpenAI Fallback)
  • Claude 카드 자동 생성 · OpenAI 임베딩 · pgvector 시맨틱 검색 · RAG(SSE)
  • 소비 note-created → 카드 생성, AiCard Consumer(DLQ·멱등)
세 번째, 노트가 카드가 되는 제품의 핵심입니다. 김나경 님의 learning-ai가 note-created 이벤트를 받아 클로드로 플래시카드를 자동 생성합니다 — 노트만 있으면 복습 자료가 완성되죠. LLM은 Tenacity 재시도와 OpenAI 폴백으로 이중화했고, 마크다운 섞인 응답에 대비한 방어적 파싱과 DLQ·멱등으로 안정성을 잡았습니다. 임베딩과 RAG도 여기서 담당합니다.

④ 카드로 복습한다 — SRS · learning-card · 조유지 (트랙 D-1)

만들어진 카드를 간격 반복으로 복습한다

  • 덱·카드 CRUD, SM-2 4버튼 복습 스케줄링
  • 복습 세션·통계 (overview/heatmap/retention)
  • 발행 review-completed·review-due, Flutter 복습 UI 연동
네 번째, 만들어진 카드로 복습하는 단계입니다. 조유지 님의 learning-card가 SM-2 4버튼으로 복습을 스케줄링합니다. 카드를 뒤집어 답을 확인하고 난이도를 고르면 알고리즘이 다음 복습일을 잡죠. 통계는 일별·히트맵·정착률로 제공하고, 복습이 끝나면 review-completed 이벤트를 발행해 다음 보상 단계로 넘어갑니다.

⑤ 복습이 보상이 된다 — engagement · 한승완 (트랙 B)

복습이 XP·레벨·배지로 환원되고 알림으로 돌아온다

  • community — 그룹 CRUD · 멤버 관리 · 공유(덱/노트)·fork · 검색 · 신고/모더레이션
  • gamification — XP · 레벨 · 배지 · 스트릭 · 리더보드(Redis 캐시 → DB 폴백)
  • 이벤트 — 소비 user-registered(프로필 생성)·review-completed(XP 적립), 발행 level-up·badge-earned·notification-send, 그룹 engagement-svc-group
  • 검증 — Step 9~11 E2E(복습→XP→레벨업→audit), 멱등성(eventId / cardId+reviewedAt)
다섯 번째, 복습이 보상이 되는 단계입니다. 한승완 님의 engagement가 review-completed를 받아 XP·레벨·배지를 부여하고, 레벨업과 배지는 다시 알림으로 환류됩니다 — 첫 단계 플랫폼으로 돌아가는 거죠. 커뮤니티 그룹·공유·모더레이션도 여기 있고, 점수 누적에도 멱등성을 보장해 중복 한 번에 리더보드가 틀어지지 않게 했습니다.

⑥ 다시 찾는다 — 검색·재발견 · knowledge · 박은서 (트랙 C-2)

쌓인 지식을 의미로 다시 꺼낸다

  • 비동기 청킹 → OpenAI 임베딩 → note_chunks(pgvector)
  • ES BM25(Nori) + RRF 하이브리드(시맨틱은 learning-ai 위임) + 키워드 폴백
  • Modulith 경계(ArchUnit) · Avro 스키마 · 검색 E2E CI 복구
여섯 번째, 쌓인 지식을 다시 찾는 단계입니다. 박은서 님의 knowledge 검색입니다. 노트를 청크로 나눠 임베딩하고, 키워드 BM25와 의미 임베딩을 RRF로 융합해 단일 랭킹을 냅니다. 시맨틱 호출이 실패하면 키워드 단독으로 폴백해 검색이 멈추지 않게 했고, ArchUnit으로 모듈 경계를, CI로 검색 E2E를 지켰습니다.

⑦ 이 모든 걸 떠받친다 — Gateway·인프라·CI/CD · 김민구 (팀장)

①~⑥ 전 여정이 도는 무대 — 게이트웨이·이벤트·운영

  • Gateway — /api/{svc}/** 라우팅(StripPrefix) · JWT 엣지검증+신원 전파 · Redis rate-limit(1rps·burst 60) · CORS · frontend 캐치올
  • 인프라(IaC) — EKS·RDS·MSK·ElastiCache·ES Terraform, ArgoCD GitOps(dev/staging/prod), bring-up 멱등 자동화(kafka-topics·db-init·es-reindex)
  • 공유·표준 — shared Avro 계약·Schema Registry, reusable CI(deploy/mirror/flyway-guard), 토픽 환경 프리픽스(#199)
  • 운영·SLA — PR 크로스리뷰, staging 16/16 Healthy · 24h 소크, SLA 실측 5/7(P1 79.7ms·P4 1.31s·P7 FCM 100%), 장애 추적·복구
마지막 일곱 번째, 이 모든 걸 떠받치는 백본입니다. 제가 맡은 게이트웨이와 인프라입니다. 게이트웨이가 JWT를 엣지에서 검증하고 신원을 전파하며 레이트리밋을 겁니다. 인프라는 EKS·RDS·MSK·ES를 테라폼으로, 배포를 ArgoCD GitOps로, 기동을 멱등 자동화로 만들었습니다. 설계로 끝내지 않고 실제 AWS에 dev·staging을 올려 16/16 정상, SLA 5/7 충족, 24시간 안정화까지 증명했습니다.
