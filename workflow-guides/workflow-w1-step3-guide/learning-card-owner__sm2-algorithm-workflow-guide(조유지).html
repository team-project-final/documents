작업 이해

이 작업은 체크리스트를 단순히 순서대로 처리하는 것이 아니라, 먼저 해당 Step이 어떤 사용자 가치와 시스템 책임을 검증하거나 구현하려는지 이해한 뒤 진행해야 한다. 시작 전에 PRD, TASK, WORKFLOW의 목표와 완료 기준을 연결해서 읽고, 이번 Step의 산출물이 이전 주차 결과물과 다음 Step에 어떤 영향을 주는지 확인한다.

처음 프로젝트를 접한 개발자는 기능명만 보고 바로 구현하지 말고, 담당 서비스의 경계, 입력/출력 계약, 보안 제약, 테스트 기준을 먼저 정리한다. 이해한 내용을 TASK의 Instructions 또는 Constraints에 반영한 다음 설계, 구현, 검증을 진행하는 것을 기본 흐름으로 삼는다.

이 문서의 목적

이 가이드는 처음 프로젝트를 접한 개발자가 WORKFLOW_learning-card_W1.md의 W1 Step 바로 진행할 수 있도록 원본 workflow 체크리스트를 실행 순서로 재구성한 문서다.

Done When 기준

아래 체크리스트가 완료되고, 관련 테스트·빌드·실행 확인 결과가 PR 산출물에 포함되면 Step 완료로 판단한다.

진행 체크리스트

1.1 TASK 시작

  • Step Goal / Done When / Scope / Input 확인
  • PRD_W1 해당 요구사항 확인 (FR-LC-xxx SM-2 복습)
  • Duration 산정 확인 (1.5일)

1.2 요구사항 분석

  • SM-2 알고리즘 논문/명세 분석
  • rating enum 정의 (Again=0, Hard=1, Good=2, Easy=3)
  • interval/EF 계산 규칙 정리
  • Instructions 초안 → TASK 문서 반영

1.3 Security 1차 검토

  • 인증 필요 여부: Yes (JWT 인증 필요)
  • 권한 종류: 로그인 사용자 (본인 카드만 복습)
  • 공개 API 여부: No
  • 결과 → TASK Constraints 반영

1.4 ERD 설계

  • review_logs 테이블 설계 (id, cardId, userId, rating, interval, easeFactor, reviewedAt)
  • 인덱스 설계 (cardId+userId, reviewedAt DESC)
  • 관계 정의 (review_logs.cardId → cards.id FK)
  • Duration(final) 갱신

1.5 Security 2차 검토

  • 민감 정보 암호화: 비해당
  • Soft Delete 정책: 물리삭제 (로그 누적 보관)
  • 행 단위 접근 제어: 필요 (본인 카드 복습 기록만)
  • 결과 → TASK Constraints 반영

1.6 DTO / Entity 설계 (API First)

  • ReviewRequest 정의 (rating)
  • ReviewResponse 정의 (nextReviewDate, interval, easeFactor)
  • ReviewLog Entity 작성
  • Rating Enum 작성 (AGAIN, HARD, GOOD, EASY)
  • MapStruct 매퍼 작성
  • Output Format → TASK 반영

1.7 Repository 구현

  • ReviewLogRepository 인터페이스 작성
  • findTopByCardIdOrderByReviewedAtDesc 커스텀 쿼리
  • Flyway 마이그레이션 스크립트 작성

1.8 Service + Test

  • Sm2Calculator 도메인 서비스 구현
  • interval 계산 로직 (Again→1, Hard→유지, Good→interval*EF, Easy→interval*EF*2)
  • easeFactor 업데이트 로직 (최소 1.3 보장)
  • 단위 테스트 작성 (4개 rating x 초기/중간/고EF 경계값)
  • 부동소수점 반올림 처리 확인
  • 테스트 통과 확인

1.9 Controller + Test

  • POST /api/v1/cards/{cardId}/review 엔드포인트 구현
  • 슬라이스 테스트 (@WebMvcTest)
  • 401/403 응답 테스트
  • 통합 테스트 (복습 → interval 변경 확인)
  • 테스트 통과 확인

1.10 View + Test (해당 시)

  • Flutter 화면 연동: 해당 없음 (프론트 별도)
  • Swagger API 문서 확인
  • RULE Reference → TASK 반영
