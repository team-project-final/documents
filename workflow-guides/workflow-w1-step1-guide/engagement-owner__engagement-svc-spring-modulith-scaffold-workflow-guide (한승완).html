1. 먼저 이해할 것

서비스 역할

engagement-svc는 커뮤니티 그룹과 게임화 경험을 담당한다. W1 Step 1에서는 기능을 만들지 않고, 이후 그룹 CRUD, 멤버 관리, 공유, 신고, XP/레벨/배지 기능이 들어갈 수 있는 모듈 기반만 만든다.

골격 단계의 보안 판단

인증과 권한은 아직 구현하지 않는다. 공개 API는 비즈니스 API가 아니라 /actuator/health 확인만 허용한다. Soft Delete와 행 단위 접근 제어는 후속 기능 단계에서 다룬다.

이 문서의 목적

이 문서는 engagement-svc를 처음 맡은 개발자가 Spring Boot 4 + Spring Modulith 기반 프로젝트 골격을 만들고, community와 gamification 모듈 경계를 검증하는 순서를 설명한다.

Done When

Spring Boot 4 프로젝트가 생성되고, 2개 모듈 패키지와 빈 Controller/Service, package-info.java, ApplicationModulesTest, multi-stage Dockerfile이 준비되며 ./gradlew build와 Docker 이미지 빌드가 통과하면 완료로 본다.

착수 전 확인

wiki 18번 기술 문서는 Spring Modulith 1.x를 Spring Boot 4 호환 기준으로 둔다. 실제 build.gradle.kts 작성 시에는 platform-svc와 동일한 Gradle 구조와 BOM 기준을 맞춘다.

WORKFLOW 진행 순서

1TASK 시작

  • Step Goal, Done When, Scope, Input을 확인한다.
  • PRD_W1에서 프로젝트 골격 요구사항을 확인한다.
  • Duration은 0.5일로 유지한다.

2요구사항 분석

  • Spring Boot 4 + Java 21 + Gradle Kotlin DSL 기준으로 생성한다.
  • community/gamification의 책임을 분리해 TASK에 반영한다.
  • platform-svc와 동일한 빌드 구조를 우선 참조한다.

3Security 1차 검토

  • 골격 단계이므로 인증은 구현하지 않는다.
  • 비즈니스 공개 API는 만들지 않는다.
  • Actuator health만 확인 대상으로 둔다.

4ERD 설계

  • DB 연동과 ERD는 이번 단계 범위 밖이다.
  • 대신 모듈별 패키지 구조도를 문서화한다.
  • Duration final을 TASK에 갱신한다.

5Security 2차 검토

  • 민감 정보 암호화는 비해당으로 기록한다.
  • Soft Delete와 행 단위 접근 제어는 후속 community 기능 단계로 넘긴다.
  • 모듈 경계는 설계 보안 항목으로 별도 관리한다.

6DTO / Entity 설계

  • DTO와 Entity는 만들지 않는다.
  • 빈 Controller/Service 클래스만 생성한다.
  • 각 모듈에 package-info.java를 작성한다.

7Repository 구현

  • Repository는 비해당으로 둔다.
  • ApplicationModulesTest를 작성한다.
  • 순환 의존과 경계 위반을 빌드에서 잡도록 한다.

8Service + Test

  • 2개 모듈에 빈 Service를 만든다.
  • ApplicationModulesTest를 실행한다.
  • ./gradlew build 성공을 확인한다.

9Controller + Test

  • 2개 모듈에 빈 Controller를 만든다.
  • multi-stage Dockerfile을 작성한다.
  • Docker 이미지 빌드 성공을 확인한다.

10View + Test

  • Flutter 연동은 비해당으로 표시한다.
  • docker compose에서 synapse-engagement-svc 실행을 확인한다.
  • RULE Reference를 TASK에 반영한다.

의존성은 초안이다

그룹 가입, 공유, 신고 같은 community 이벤트가 XP나 배지 계산으로 이어질 수 있으므로 골격 단계에서는 gamification이 community의 명시 public API 또는 이벤트만 참조하는 방향으로 시작한다. 직접 호출이 필요 없으면 allowedDependencies를 비워 더 엄격하게 관리한다.

Docker 구성 예시

Dockerfile

FROM eclipse-temurin:21-jdk AS builder
WORKDIR /workspace
COPY gradlew settings.gradle.kts build.gradle.kts ./
COPY gradle ./gradle
COPY src ./src
RUN ./gradlew clean bootJar --no-daemon

FROM eclipse-temurin:21-jre
WORKDIR /app
COPY --from=builder /workspace/build/libs/*.jar app.jar
EXPOSE 8082
ENTRYPOINT ["java", "-jar", "/app/app.jar"]

docker compose 서비스 조각

synapse-engagement-svc:
  build:
    context: ./engagement-svc
  container_name: synapse-engagement-svc
  ports:
    - "8082:8082"
  environment:
    SPRING_PROFILES_ACTIVE: local
  healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost:8082/actuator/health"]
    interval: 10s
    timeout: 3s
    retries: 5

검증 명령

# 1. 테스트 포함 빌드
./gradlew clean build

# 2. Modulith 구조 검증 테스트만 실행
./gradlew test --tests "*ApplicationModulesTest"

# 3. 로컬 실행
./gradlew bootRun

# 4. Health endpoint 확인
curl http://localhost:8082/actuator/health

# 5. Docker 이미지 빌드
docker build -t synapse-engagement-svc:local .

# 6. Docker 컨테이너 실행 후 health 확인
docker run --rm -p 8082:8082 synapse-engagement-svc:local
curl http://localhost:8082/actuator/health
수동 위반 검증

검증용으로만 다른 모듈의 internal 타입을 import하거나 순환 의존을 만든 뒤 ApplicationModules.verify()가 실패하는지 확인한다. 확인 후 위반 코드는 반드시 제거한다.
