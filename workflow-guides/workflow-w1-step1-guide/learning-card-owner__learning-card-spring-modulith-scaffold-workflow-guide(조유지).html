작업 이해

이 작업은 체크리스트를 단순히 순서대로 처리하는 것이 아니라, 먼저 해당 Step이 어떤 사용자 가치와 시스템 책임을 검증하거나 구현하려는지 이해한 뒤 진행해야 한다. 시작 전에 PRD, TASK, WORKFLOW의 목표와 완료 기준을 연결해서 읽고, 이번 Step의 산출물이 이전 주차 결과물과 다음 Step에 어떤 영향을 주는지 확인한다.

처음 프로젝트를 접한 개발자는 기능명만 보고 바로 구현하지 말고, 담당 서비스의 경계, 입력/출력 계약, 보안 제약, 테스트 기준을 먼저 정리한다. 이해한 내용을 TASK의 Instructions 또는 Constraints에 반영한 다음 설계, 구현, 검증을 진행하는 것을 기본 흐름으로 삼는다.

1. 먼저 이해할 것

이 작업은 덱/카드 CRUD나 SM-2 알고리즘을 구현하는 단계가 아니라, learning-card 서비스가 빌드되고 Modulith 검증과 Health endpoint가 동작하는 골격을 만드는 단계다. 이후 card 모듈과 srs 모듈이 독립적으로 커질 수 있도록 패키지 경계를 먼저 고정한다.

완료 기준

./gradlew build 성공, ApplicationModules.verify() 통과, /actuator/health 응답, Docker 이미지 빌드 성공, docker compose에서 learning-card 서비스 실행 확인이다.

착수 전 포트 확인

TASK_learning-card는 포트 8082를 지정한다. 반면 환경 설정/배포 문서에는 synapse-learning-card가 8084로 등장한다. 실제 구현 전 팀 공통 포트를 확정하고 TASK 또는 환경 문서에 반영한다.

3. 목표 패키지 구조

기본 패키지는 예시로 com.synapse.learningcard를 사용한다. 실제 생성된 Spring Initializr group/package에 맞게 치환한다.

learning-card/ ├── build.gradle.kts ├── Dockerfile ├── src/main/java/com/synapse/learningcard/ │ ├── LearningCardApplication.java │ ├── card/ │ │ ├── package-info.java │ │ ├── CardController.java │ │ └── CardService.java │ └── srs/ │ ├── package-info.java │ ├── SrsController.java │ └── SrsService.java ├── src/main/resources/ │ └── application.yml └── src/test/java/com/synapse/learningcard/ └── ApplicationModulesTest.java
골격 단계 원칙

Controller와 Service는 빈 클래스 또는 최소 Health 외 확인 가능한 수준만 만든다. Entity, Repository, Flyway 마이그레이션, 비즈니스 API는 Step 2 이후에 추가한다.

4. 모듈 역할 정의

모듈 W1 이후 책임 골격 단계 산출물 의존성 기준
card Deck/Card CRUD, 카드 조회, 수동 카드 관리, 이후 Kafka 이벤트 발행 지점 package-info.java, 빈 CardService, 빈 CardController 초기에는 srs에 직접 의존하지 않음
srs SM-2 계산, 복습 세션, due date 계산, 복습 통계 기반 package-info.java, 빈 SrsService, 빈 SrsController 초기에는 card public API만 필요 시 허용

5. 진행 순서

Step 1. TASK 시작

1.1
  • Step Goal / Done When / Scope / Input을 TASK_learning-card에서 확인한다.
  • PRD_W1의 프로젝트 골격 및 learning-card 요구사항을 확인한다.
  • Duration 0.5일 기준으로 비즈니스 로직을 제외한다.

Step 2. 요구사항 분석

1.2
  • Spring Boot 4 + Modulith 프로젝트 구조를 분석한다.
  • card/srs 2개 모듈 역할을 표로 정의한다.
  • 팀 공통 Gradle 설정과 포트 정책을 확인한다.
  • Instructions 초안을 TASK 또는 PR 설명에 반영한다.

Step 3. Security 1차 검토

1.3
  • 골격만 생성하므로 인증 필요 여부는 No로 기록한다.
  • 권한 종류는 없음으로 기록한다.
  • 공개 API는 기능 API가 아니라 Health endpoint만으로 기록한다.
  • 결과를 TASK Constraints에 반영한다.

Step 4. ERD 설계

1.4
  • 골격 단계라 ERD는 해당 없음으로 처리한다.
  • card/srs 모듈별 패키지 구조도를 작성한다.
  • 실제 작업량을 보고 Duration final을 갱신한다.

Step 5. Security 2차 검토

1.5
  • 민감 정보 암호화는 비해당으로 기록한다.
  • Soft Delete 정책은 Step 2 CRUD부터 다루므로 비해당으로 기록한다.
  • 행 단위 접근 제어도 골격 단계에서는 불필요로 기록한다.
  • 결과를 TASK Constraints에 반영한다.

Step 6. DTO / Entity 설계

1.6
  • 골격 단계라 빈 Controller/Service 클래스만 생성한다.
  • 각 모듈에 package-info.java와 @ApplicationModule을 설정한다.
  • Output Format에 구조도, 빌드 로그, Health 응답을 반영한다.

Step 7. Repository 구현

1.7
  • Repository는 골격 단계에서 해당 없음으로 처리한다.
  • ApplicationModulesTest 구조 검증 테스트를 작성한다.

Step 8. Service + Test

1.8
  • card, srs 모듈에 빈 Service 클래스를 생성한다.
  • ApplicationModulesTest 통과를 확인한다.
  • ./gradlew build 성공을 확인한다.
  • /actuator/health 응답을 확인한다.

Step 9. Controller + Docker

1.9
  • card, srs 모듈에 빈 Controller 클래스를 생성한다.
  • multi-stage Dockerfile을 작성한다.
  • Docker 이미지 빌드 성공을 확인한다.

Step 10. View + Test

1.10
  • Flutter 화면 연동은 해당 없음으로 처리한다.
  • docker compose에서 learning-card-svc 실행을 확인한다.
  • RULE Reference를 TASK 또는 PR 설명에 반영한다.

왜 빈 클래스인가

9. Docker와 Compose 예시

Dockerfile

FROM eclipse-temurin:21-jdk AS build
WORKDIR /workspace
COPY gradlew settings.gradle.kts build.gradle.kts ./
COPY gradle ./gradle
COPY src ./src
RUN ./gradlew bootJar --no-daemon

FROM eclipse-temurin:21-jre
WORKDIR /app
COPY --from=build /workspace/build/libs/*.jar app.jar
EXPOSE 8082
ENTRYPOINT ["java", "-jar", "app.jar"]

docker-compose.yml 추가

services:
  learning-card-svc:
    build:
      context: ./learning-card
    container_name: synapse-learning-card
    ports:
      - "8082:8082"
    environment:
      SPRING_PROFILES_ACTIVE: local
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8082/actuator/health"]
      interval: 10s
      timeout: 3s
      retries: 5
