1. 먼저 이해할 것

서비스 역할

platform-svc는 인증, 사용자, 알림, 운영 관리의 기반 서비스다. W1 Step 1에서는 OAuth, JWT, DB, 외부 서비스 연동을 만들지 않고, 후속 구현을 받을 수 있는 모듈 구조와 빌드 기반만 만든다.

골격 단계의 보안 판단

인증 서비스이지만 이번 단계에서는 인증을 구현하지 않는다. 공개 API는 비즈니스 API가 아니라 /actuator/health 확인만 허용한다. 민감 정보, Soft Delete, 행 단위 접근 제어는 후속 기능 단계에서 다룬다.

이 문서의 목적

이 문서는 platform-svc를 처음 맡은 개발자가 Spring Boot 4 + Spring Modulith 기반 프로젝트 골격을 만들고, auth, user, notification, admin 모듈 경계를 검증하는 순서를 설명한다.

Done When

Spring Boot 4 프로젝트가 생성되고, 4개 모듈 패키지와 빈 Controller/Service, package-info.java, ApplicationModulesTest, multi-stage Dockerfile이 준비되며 ./gradlew build와 Docker 이미지 빌드가 통과하면 완료로 본다.

착수 전 확인

TASK/WORKFLOW는 auth/user/notification/admin 4개 모듈을 기준으로 한다. 다만 wiki 03/18에는 platform-svc가 auth/audit/billing/notification으로도 등장한다. 실제 생성 전 팀 공통 모듈명을 확정하고, 이 문서는 현재 WORKFLOW 기준으로 진행한다.

WORKFLOW 진행 순서

1TASK 시작

  • Step Goal, Done When, Scope, Input을 확인한다.
  • PRD_W1에서 프로젝트 골격 요구사항을 확인한다.
  • Duration은 1일로 확인한다.

2요구사항 분석

  • Spring Boot 4 + Java 21 + Gradle Kotlin DSL 기준으로 생성한다.
  • auth/user/notification/admin의 책임을 분리해 TASK에 반영한다.
  • Gradle 의존성 목록을 먼저 확정한다.

3Security 1차 검토

  • 골격 단계이므로 인증은 구현하지 않는다.
  • 비즈니스 공개 API는 만들지 않는다.
  • Actuator health만 확인 대상으로 둔다.

4ERD 설계

  • DB 마이그레이션은 Out of Scope다.
  • 모듈별 패키지 구조도를 문서화한다.
  • Duration final을 TASK에 갱신한다.

5Security 2차 검토

  • 민감 정보 암호화는 비해당으로 기록한다.
  • Soft Delete와 행 단위 접근 제어는 후속 기능 단계로 넘긴다.
  • 모듈 경계는 설계 보안 항목으로 별도 관리한다.

6DTO / Entity 설계

  • DTO와 Entity는 만들지 않는다.
  • 빈 Controller/Service 클래스만 생성한다.
  • 각 모듈에 package-info.java를 작성한다.

7Repository 구현

  • Repository는 비해당으로 둔다.
  • ApplicationModulesTest를 작성한다.
  • 순환 의존과 경계 위반을 빌드에서 잡도록 한다.

8Service + Test

  • 4개 모듈에 빈 Service를 만든다.
  • ApplicationModulesTest를 실행한다.
  • ./gradlew build 성공을 확인한다.

9Controller + Test

  • 4개 모듈에 빈 Controller를 만든다.
  • multi-stage Dockerfile을 작성한다.
  • Docker 이미지 빌드 성공을 확인한다.

10View + Test

  • Flutter 연동은 비해당으로 표시한다.
  • docker compose에서 synapse-platform-svc 실행을 확인한다.
  • RULE Reference를 TASK에 반영한다.

골격 단계 의존성 기준

OAuth, Security, Data JPA, Redis, Kafka, Mail, FCM/APNs, Stripe 등은 후속 Step에서 필요할 때 추가한다. Step 1에서는 Web, Actuator, Modulith, Test만으로 빌드 시간을 짧게 유지한다.

package-info.java

// src/main/java/com/synapse/platform/auth/package-info.java
@ApplicationModule(
    displayName = "Auth",
    allowedDependencies = {"user"}
)
package com.synapse.platform.auth;

import org.springframework.modulith.ApplicationModule;
// src/main/java/com/synapse/platform/user/package-info.java
@ApplicationModule(
    displayName = "User",
    allowedDependencies = {}

package com.synapse.platform.user;

import org.springframework.modulith.ApplicationModule;
// src/main/java/com/synapse/platform/notification/package-info.java
@ApplicationModule(
    displayName = "Notification",

import org.springframework.modulith.ApplicationModule;

import org.springframework.modulith.ApplicationModule;
의존성은 초안이다

user를 기반 모듈로 두고 auth/notification/admin이 명시 API만 참조하는 구조로 시작한다. 이벤트 기반으로 충분하면 allowedDependencies를 더 줄인다.

Docker 구성 예시

Dockerfile

FROM eclipse-temurin:21-jdk AS builder
WORKDIR /workspace
COPY gradlew settings.gradle.kts build.gradle.kts ./
COPY gradle ./gradle
COPY src ./src
RUN ./gradlew clean bootJar --no-daemon

FROM eclipse-temurin:21-jre
WORKDIR /app
COPY --from=builder /workspace/build/libs/*.jar app.jar
EXPOSE 8081
ENTRYPOINT ["java", "-jar", "/app/app.jar"]

docker compose 서비스 조각

synapse-platform-svc:
  build:
    context: ./platform-svc
  container_name: synapse-platform-svc
  ports:
    - "8081:8081"
  environment:
    SPRING_PROFILES_ACTIVE: local
  healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost:8081/actuator/health"]
    interval: 10s
    timeout: 3s
    retries: 5

검증 명령

# 1. 테스트 포함 빌드
./gradlew clean build

# 2. Modulith 구조 검증 테스트만 실행
./gradlew test --tests "*ApplicationModulesTest"

# 3. 로컬 실행
./gradlew bootRun

# 4. Health endpoint 확인
curl http://localhost:8081/actuator/health

# 5. Docker 이미지 빌드
docker build -t synapse-platform-svc:local .

# 6. Docker 컨테이너 실행 후 health 확인
docker run --rm -p 8081:8081 synapse-platform-svc:local
curl http://localhost:8081/actuator/health
수동 위반 검증

검증용으로만 다른 모듈의 internal 타입을 import하거나 순환 의존을 만든 뒤 ApplicationModules.verify()가 실패하는지 확인한다. 확인 후 위반 코드는 반드시 제거한다.
