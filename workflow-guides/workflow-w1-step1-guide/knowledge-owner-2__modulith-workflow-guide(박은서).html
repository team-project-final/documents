작업 이해

이 작업은 체크리스트를 단순히 순서대로 처리하는 것이 아니라, 먼저 해당 Step이 어떤 사용자 가치와 시스템 책임을 검증하거나 구현하려는지 이해한 뒤 진행해야 한다. 시작 전에 PRD, TASK, WORKFLOW의 목표와 완료 기준을 연결해서 읽고, 이번 Step의 산출물이 이전 주차 결과물과 다음 Step에 어떤 영향을 주는지 확인한다.

처음 프로젝트를 접한 개발자는 기능명만 보고 바로 구현하지 말고, 담당 서비스의 경계, 입력/출력 계약, 보안 제약, 테스트 기준을 먼저 정리한다. 이해한 내용을 TASK의 Instructions 또는 Constraints에 반영한 다음 설계, 구현, 검증을 진행하는 것을 기본 흐름으로 삼는다.

1. 먼저 이해할 것

이 작업은 기능 구현이 아니라 knowledge-svc 안의 모듈 경계와 검증 장치를 만드는 설정 작업이다. 노트 CRUD, 그래프 알고리즘, 청크 분할 로직을 구현하기 전에 각 모듈이 어디까지 접근할 수 있는지 정하고, 그 규칙을 빌드에서 자동 검증되게 만드는 것이 목표다.

완료 기준

ApplicationModules.verify()가 통과하고, 모듈 간 금지된 직접 import 또는 internal 패키지 접근이 들어오면 테스트/빌드에서 실패해야 한다.

착수 전 버전 확인

18번 기술 스택 문서는 Spring Boot 4, Java 21, Spring Modulith 1.x를 명시한다. 반면 TASK_knowledge-2의 Constraints에는 Spring Modulith 2.x가 적혀 있다. 실제 build.gradle.kts를 작성하기 전 팀장 또는 tech lead에게 버전을 확정받고, 확정 결과를 TASK 문서에 반영한다.

3. 목표 구조

기본 패키지는 예시로 com.synapse.knowledge를 사용한다. 실제 프로젝트 생성 시 루트 패키지가 다르면 같은 원칙으로 치환한다.

src/main/java/com/synapse/knowledge/ ├── KnowledgeApplication.java ├── note/ │ ├── package-info.java │ ├── api/ │ │ ├── NoteReader.java │ │ └── NoteSummary.java │ └── internal/ │ ├── NoteService.java │ ├── NoteRepository.java │ └── NoteInternalMapper.java ├── graph/ │ ├── package-info.java │ ├── api/ │ │ └── GraphLinkUpdater.java │ └── internal/ │ └── GraphService.java └── chunking/ ├── package-info.java ├── api/ │ └── ChunkingRequestPort.java └── internal/ └── ChunkingService.java
패키지 규칙

각 모듈의 외부 공개 지점은 루트 패키지 또는 api 패키지로 제한한다. 구현체, JPA 엔티티, Repository, Mapper, 내부 Service는 internal 아래 둔다. 다른 모듈은 internal을 import하지 않는다.

4. 권장 의존성 방향

모듈 간 호출은 최소화하고, 필요하면 public API 인터페이스 또는 애플리케이션 이벤트로 통신한다. W1의 목적은 “향후 분리 가능한 구조”를 만드는 것이므로 직접 구현체 호출을 기본값으로 두지 않는다.

모듈 책임 허용 의존성 초안 금지할 접근
note Markdown CRUD, 위키링크 파싱, 노트 이벤트 발행 가능하면 없음. 그래프 갱신은 이벤트 또는 graph::api로 제한 graph.internal, chunking.internal 직접 접근
graph 백링크, 노드/엣지 데이터, graph.notes.linked 이벤트 note::api만 허용 note.internal.NoteRepository 직접 접근
chunking 비동기 청크 분할, 임베딩 요청 준비 note::api만 허용 note.internal 직접 접근, 동기식 note 저장 흐름 지연
의존성 결정 기준

note 저장 후 graph/chunking 처리가 필요하면 먼저 Spring Modulith 이벤트를 검토한다. 같은 JVM 안에서 즉시 조회가 필요한 경우에만 public API 인터페이스를 열고, 그 인터페이스를 allowedDependencies에 반영한다.

5. 작업 순서

Step 1. TASK 시작

1.1
  • Step Goal / Done When / Scope / Input을 TASK_knowledge-2에서 확인한다.
  • PRD_W1의 FR-K2-001, FR-K2-002를 수용 기준으로 표시한다.
  • Duration 1일 안에 끝낼 수 있도록 비즈니스 로직 구현을 제외한다.

Step 2. 요구사항 분석

1.2
  • Spring Modulith의 application module, named interface, verification 개념을 확인한다.
  • note / graph / chunking 의존성 방향을 표로 정리한다.
  • public API 인터페이스는 다른 모듈이 실제로 필요로 하는 읽기/명령만 노출한다.
  • 확정한 지침을 TASK 문서 또는 PR 설명에 남긴다.

Step 3. Security 1차 검토

1.3
  • 이 작업 자체는 설정 작업이므로 인증/권한/공개 API를 만들지 않는다.
  • 보안 관점의 핵심은 internal 패키지 외부 접근 차단이다.
  • 컴파일 또는 테스트에서 경계 위반을 발견하도록 검증을 추가한다.
  • 결과를 TASK Constraints에 “public API 외 접근 금지”로 반영한다.

Step 4. 모듈 정의 및 구조 설계

1.4
  • note, graph, chunking 패키지를 만든다.
  • 각 모듈 루트에 package-info.java를 작성한다.
  • @ApplicationModule(allowedDependencies = ...)를 설정한다.
  • api와 internal 패키지를 분리한다.
  • 모듈 구조 다이어그램을 README 또는 TASK 산출물에 추가한다.

Step 5. 모듈 경계 검증

1.5
  • ApplicationModules.verify() 통합 테스트를 작성한다.
  • 의도적으로 잘못된 import를 넣어 실패를 확인한 뒤 제거한다.
  • 순환 의존이 검출되는지 확인한다.
  • internal 접근 차단은 Modulith 검증과 ArchUnit Step 2에서 함께 보강한다.

7. public API 설계 기준

public API는 “다른 모듈이 알아도 되는 계약”이다. 구현체나 저장소가 아니라 인터페이스, command/query DTO, event만 둔다. W1에서는 빈 골격이어도 좋지만 이름은 이후 기능 구현자가 자연스럽게 사용할 수 있어야 한다.

API 후보 위치 용도 주의
NoteReader note.api graph/chunking이 노트 요약 정보를 읽어야 할 때 사용 JPA Entity를 반환하지 말고 NoteSummary 같은 읽기 DTO 반환
GraphLinkUpdater graph.api note가 위키링크 변경을 graph에 알려야 할 때 사용 가능하면 동기 호출보다 이벤트 기반 처리 우선
ChunkingRequestPort chunking.api 노트 저장 후 청크 분할 요청을 enqueue 노트 저장 트랜잭션을 지연시키지 않도록 비동기 전제

8. 검증 방법

  1. ./gradlew test를 실행해 ApplicationModulesTest 통과를 확인한다.
  2. graph.internal에서 note.internal.NoteRepository를 import하는 임시 위반 코드를 만든다.
  3. 테스트가 실패하는지 확인하고 실패 로그를 캡처한다.
  4. 임시 위반 코드를 제거하고 다시 테스트를 통과시킨다.
  5. PR에는 PASS 로그와 FAIL 확인 사실을 함께 남긴다.
금지

실패 검증을 위해 만든 위반 코드를 커밋하지 않는다. PR에는 “어떤 위반을 넣었고 어떤 검증이 실패했는지”만 로그나 설명으로 남긴다.
