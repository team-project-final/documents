WORKFLOW Guide - W5 Step 1: 단위 테스트 보강 — 핵심 도메인/서비스 로직 검증

5주차에는 별도 원본 WORKFLOW 문서가 없으므로, W1~W4 문서와 가이드에서 구현된 기능을 기준으로 테스트 중심의 실행 가이드를 구성한다. 이 문서는 knowledge-svc note/graph 담당자가 바로 검증 항목을 정리하고 실행할 수 있게 만든다.

Step 1대상 단계

진행 기준

W1~W4에서 구현한 기능의 핵심 로직을 단위 테스트로 고정하고, 리팩터링이나 버그 수정 중 회귀가 즉시 드러나게 한다. 테스트 결과는 단순 통과 여부만 남기지 말고, 실행 명령, 환경, 실패 재현 절차, 수정 후 재검증 범위까지 TASK 또는 HISTORY에 연결한다.

공통 위험

  • W1~W4 구현 범위와 실제 테스트 범위가 어긋나면 릴리즈 판단이 왜곡된다.
  • 외부 연동 테스트는 비용, rate limit, 테스트 데이터 오염을 피하도록 mock/fake 기준을 명확히 한다.
  • flaky test는 통과로 처리하지 말고 원인과 격리 여부를 별도 기록한다.

산출물

  • 테스트 실행 명령과 결과 요약
  • 실패 케이스 재현 절차와 담당자
  • 커버리지 또는 시나리오 매핑표
  • 릴리즈 차단/비차단 결함 목록

1. TASK 시작

  • W1~W4 TASK/WORKFLOW에서 구현 완료된 기능 목록 확인
  • 단위 테스트 대상 함수, 서비스, 유틸, 정책 로직 목록화
  • Done When: 핵심 분기/예외/경계값 테스트가 통과하고 CI에서 반복 실행 가능함

2. 테스트 범위 분석

  • 도메인 규칙, 권한 판정, 상태 전이, 날짜/시간 계산, 이벤트 페이로드 생성 로직을 우선 선정
  • 외부 시스템 DB/Kafka/FCM/LLM/OpenSearch 호출은 mock 또는 fake로 격리
  • 과거 W1~W4 가이드의 Security Constraints가 테스트 케이스에 반영됐는지 확인

트랙별 중점 검증

  • note wikilink parsing, graph backlink, version restore, tag filtering 테스트
  • 권한 필터와 ES 동기화 이벤트 생성 로직 테스트
  • note, graph, 버전 이력, 태그, 백링크, ES 동기화 흐름을 검증한다.

3. 단위 테스트 구현

  • 성공 케이스, 실패 케이스, 권한 오류, 중복 요청, 빈 결과, 경계값 케이스 작성
  • 테스트 이름은 Given/When/Then 또는 동작 중심으로 실패 원인이 보이게 작성
  • 랜덤, 현재 시각, 외부 응답은 고정 가능한 fixture로 분리

4. 품질 기준

  • 핵심 서비스/도메인 로직의 라인 커버리지보다 분기 커버리지와 회귀 방지력을 우선
  • 테스트가 구현 세부사항보다 공개 메서드의 관찰 가능한 결과를 검증하는지 확인
  • 깨지기 쉬운 snapshot/문자열 전체 비교는 필요한 필드 중심 검증으로 조정

5. 완료 정리

  • 로컬 테스트 명령과 CI 실행 결과 기록
  • 부족한 테스트 범위는 TODO가 아니라 이슈 또는 TASK 후속 항목으로 분리
  • RULE Reference에 테스트 기준과 실행 명령 반영

완료 전 점검

  • W1~W4 기능 중 미검증 항목이 있는지 확인하고 audit 문서에 남긴다.
  • 실패 항목은 수정 완료, 릴리즈 보류, 후속 개선 중 하나로 분류한다.
  • CI 또는 수동 테스트 결과를 재현 가능한 형태로 남긴다.
  • Step 3 종료 시 릴리즈 가능 여부와 잔여 리스크를 팀 공통 문서에 반영한다.
