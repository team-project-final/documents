1. 먼저 이해할 것

이 작업은 개별 비즈니스 기능보다 팀 전체 개발/배포 흐름을 지탱하는 운영 기반을 다루는 작업이다. 리소스를 만드는 것에서 끝내지 말고 네트워크, 권한, secret, 배포 경로, 관측 지표가 이후 서비스 개발자가 반복해서 사용할 수 있는 기준으로 남는지 확인해야 한다.

따라서 W3 Step 8을 시작할 때는 제목의 기능명만 보고 바로 코드를 작성하지 말고, 원본 WORKFLOW의 요구사항 분석, Security 검토, DTO/Entity/API 설계, 테스트 항목을 먼저 연결해서 읽는다. 그 이해를 바탕으로 TASK 문서의 Instructions, Constraints, Output Format을 갱신한 다음 구현과 검증을 진행한다.

진행 기준

TASK 문서의 Step Goal, Done When, Scope, Input을 먼저 확인하고 PRD_W3 요구사항과 불일치하는 항목은 TASK Constraints에 반영한다. 구현 전 API, 이벤트, 저장 구조를 고정하고 테스트 통과 및 문서 반영까지 완료 기준으로 삼는다.

보안/권한 체크

  • ArgoCD RBAC 설정 확인 (dev: 자동, staging: 승인 필요)
  • 배포 시 시크릿 주입 방식 확인 (Sealed Secrets / External Secrets)
  • 환경별 접근 권한 분리 확인
  • 결과 → TASK Constraints 반영
  • 환경별 환경변수/시크릿 분리 확인
  • staging 배포 승인 권한자 지정
  • 배포 이력 추적 (audit trail) 확인
  • 결과 → TASK Constraints 반영

설계 우선순위

  • 요청/응답, 저장 구조, 이벤트 형식을 구현 전에 고정한다.

1.1 TASK 시작

  • Step Goal / Done When / Scope / Input 확인
  • PRD_W3 해당 요구사항 확인 (배포 검증)
  • Duration 산정 확인

1.2 요구사항 분석

  • dev 환경 autoSync 설정 요건 분석
  • staging 환경 수동 승인 배포 플로우 분석
  • 배포 후 헬스체크 기준 정의
  • Instructions 초안 → TASK 문서 반영

1.3 Security 1차 검토

  • ArgoCD RBAC 설정 확인 (dev: 자동, staging: 승인 필요)
  • 배포 시 시크릿 주입 방식 확인 (Sealed Secrets / External Secrets)
  • 환경별 접근 권한 분리 확인
  • 결과 → TASK Constraints 반영

1.4 배포 전략 설계

  • dev 환경 ArgoCD autoSync 정책 설정
  • staging 환경 수동 승인 워크플로우 설계
  • 롤백 절차 정의 (자동/수동)
  • Duration(final) 갱신

1.5 Security 2차 검토

  • 환경별 환경변수/시크릿 분리 확인
  • staging 배포 승인 권한자 지정
  • 배포 이력 추적 (audit trail) 확인
  • 결과 → TASK Constraints 반영

1.6 배포 설정 구현

  • ArgoCD Application 매니페스트 갱신 (dev autoSync: true)
  • staging Application 매니페스트 갱신 (syncPolicy: manual)
  • 환경별 values.yaml 분리 확인

1.7 배포 실행

  • dev 환경: main push → autoSync 자동 배포 확인
  • staging 환경: 수동 Sync 버튼 → 배포 실행
  • ECR 이미지 태그 일치 확인

1.8 배포 후 검증

  • dev 환경 전체 서비스 Health OK 확인
  • staging 환경 전체 서비스 Health OK 확인
  • Kafka 연결 상태 확인 (각 서비스 consumer group lag = 0)
  • RDS/Redis/OpenSearch 연결 상태 확인

1.9 배포 이슈 대응

  • 배포 실패 시 롤백 절차 실행 및 검증
  • 환경별 로그 수집 및 이슈 분석
  • 배포 성공 기준 충족 여부 최종 확인

1.10 결과 정리

  • 배포 검증 결과 리포트 작성
  • staging 배포 승인 프로세스 문서화
  • RULE Reference → TASK 반영

완료 전 점검

  • 원본 WORKFLOW의 모든 체크 항목이 TASK 또는 구현 산출물에 대응되는지 확인한다.
  • 보안 1차/2차 검토 결과가 Constraints에 남아 있는지 확인한다.
  • 해당 단계에 필요한 단위, 슬라이스, 통합, E2E 테스트 통과 여부를 기록한다.
  • Swagger, 이벤트 스키마, 운영 가이드 등 문서 산출물이 있으면 RULE Reference에 연결한다.
