WORKFLOW Guide - W5 Step 2: 서비스 통합 테스트 — API/DB/Kafka/외부 연동 검증

5주차에는 별도 원본 WORKFLOW 문서가 없으므로, W1~W4 문서와 가이드에서 구현된 기능을 기준으로 테스트 중심의 실행 가이드를 구성한다. 이 문서는 platform-svc 담당자가 바로 검증 항목을 정리하고 실행할 수 있게 만든다.

Step 2대상 단계

진행 기준

서비스별 API, DB, Kafka, 검색, 알림, AI 연동을 실제 구성에 가깝게 검증해 단위 테스트로 잡히지 않는 계약 위반을 찾는다. 테스트 결과는 단순 통과 여부만 남기지 말고, 실행 명령, 환경, 실패 재현 절차, 수정 후 재검증 범위까지 TASK 또는 HISTORY에 연결한다.

공통 위험

  • W1~W4 구현 범위와 실제 테스트 범위가 어긋나면 릴리즈 판단이 왜곡된다.
  • 외부 연동 테스트는 비용, rate limit, 테스트 데이터 오염을 피하도록 mock/fake 기준을 명확히 한다.
  • flaky test는 통과로 처리하지 말고 원인과 격리 여부를 별도 기록한다.

산출물

  • 테스트 실행 명령과 결과 요약
  • 실패 케이스 재현 절차와 담당자
  • 커버리지 또는 시나리오 매핑표
  • 릴리즈 차단/비차단 결함 목록

1. TASK 시작

  • 서비스별 W1~W4 주요 API, 이벤트, 배치, 스케줄러 목록 확인
  • 통합 테스트 환경 요구사항 Docker Compose, Testcontainers, localstack, mock server 정리
  • Done When: API/DB/Kafka/외부 연동 경로가 테스트로 검증되고 재현 절차가 문서화됨

2. 계약/데이터 준비

  • 요청/응답 DTO, Avro/JSON Schema, DB 마이그레이션, 인덱스 매핑 버전 확인
  • 테스트 fixture는 사용자, 노트, 카드, 결제, 알림, 권한 상태를 최소 데이터로 구성
  • 토픽명, 파티션 키, 이벤트 중복 처리, 재시도 정책을 테스트 관점에서 확인

트랙별 중점 검증

  • auth/billing/notification/admin API 통합 테스트
  • RDS, Kafka audit event, FCM/SES mock 연동 검증
  • auth, user, billing, notification, audit, admin 흐름을 검증한다.

3. 통합 테스트 구현

  • Controller/API 테스트는 인증, 권한, validation, paging, error format을 포함
  • Repository/DB 테스트는 제약조건, 인덱스 조회, 트랜잭션 롤백, soft delete 조건을 포함
  • Kafka/검색/알림/AI 연동은 발행, 소비, 실패, 재시도, idempotency를 포함

4. 환경 검증

  • 로컬과 CI에서 같은 명령으로 실행 가능한지 확인
  • 테스트 데이터 격리와 cleanup이 보장되는지 확인
  • 외부 API 비용 또는 rate limit이 발생하지 않도록 mock/fake 기준 확인

5. 완료 정리

  • 통합 테스트 실행 명령, 필요한 환경변수, 실패 시 점검 순서 작성
  • 계약 위반이 발견되면 원본 PRD/TASK/WORKFLOW 중 어느 문서 기준인지 기록
  • 테스트 결과와 남은 위험을 HISTORY 또는 TASK에 반영

완료 전 점검

  • W1~W4 기능 중 미검증 항목이 있는지 확인하고 audit 문서에 남긴다.
  • 실패 항목은 수정 완료, 릴리즈 보류, 후속 개선 중 하나로 분류한다.
  • CI 또는 수동 테스트 결과를 재현 가능한 형태로 남긴다.
  • Step 3 종료 시 릴리즈 가능 여부와 잔여 리스크를 팀 공통 문서에 반영한다.
