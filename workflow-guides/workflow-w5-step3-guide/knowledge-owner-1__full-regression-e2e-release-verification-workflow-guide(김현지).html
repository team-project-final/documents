WORKFLOW Guide - W5 Step 3: 전체 회귀/E2E 테스트 — 릴리즈 후보 검증

5주차에는 별도 원본 WORKFLOW 문서가 없으므로, W1~W4 문서와 가이드에서 구현된 기능을 기준으로 테스트 중심의 실행 가이드를 구성한다. 이 문서는 knowledge-svc note/graph 담당자가 바로 검증 항목을 정리하고 실행할 수 있게 만든다.

진행 기준

사용자 관점의 핵심 시나리오를 끝까지 실행해 릴리즈 후보가 W1~W4 요구사항을 만족하는지 확인한다. 테스트 결과는 단순 통과 여부만 남기지 말고, 실행 명령, 환경, 실패 재현 절차, 수정 후 재검증 범위까지 TASK 또는 HISTORY에 연결한다.

공통 위험

  • W1~W4 구현 범위와 실제 테스트 범위가 어긋나면 릴리즈 판단이 왜곡된다.
  • 외부 연동 테스트는 비용, rate limit, 테스트 데이터 오염을 피하도록 mock/fake 기준을 명확히 한다.
  • flaky test는 통과로 처리하지 말고 원인과 격리 여부를 별도 기록한다.

산출물

  • 테스트 실행 명령과 결과 요약
  • 실패 케이스 재현 절차와 담당자
  • 커버리지 또는 시나리오 매핑표
  • 릴리즈 차단/비차단 결함 목록

1. TASK 시작

  • W1~W4 전체 기능을 사용자 여정 기준으로 묶은 E2E 시나리오 확인
  • 릴리즈 후보 버전, 배포 환경, 테스트 계정, 테스트 데이터 초기 상태 확정
  • Done When: 핵심 시나리오가 통과하고 실패/보류 항목의 릴리즈 영향도가 정리됨

2. 전체 시나리오 설계

  • 회원/인증, 노트 작성, 검색, 카드 생성, 복습, 게이미피케이션, 공유, 알림, 관리자 처리 흐름 연결
  • happy path뿐 아니라 권한 없음, 데이터 없음, 중복 요청, 네트워크 실패, 지연 응답을 포함
  • 프론트, API Gateway, 각 서비스, Kafka, DB, 검색, 알림, AI 의존성을 시나리오별로 표시

트랙별 중점 검증

  • 노트 작성-수정-버전복원-그래프 반영-검색 노출까지 E2E 검증
  • note, graph, 버전 이력, 태그, 백링크, ES 동기화 흐름을 검증한다.

3. E2E/회귀 테스트 실행

  • 자동화 가능한 화면/API 흐름은 Playwright, Flutter integration test, REST client, smoke script로 실행
  • 수동 검증 항목은 기대 결과, 실제 결과, 캡처, 로그 위치를 남김
  • 성능 SLA, 에러율, 주요 대시보드 지표를 테스트 실행 시간대와 함께 기록

4. 결함 처리

  • 결함은 재현 절차, 기대/실제 결과, 영향 범위, 우선순위, 담당자를 포함해 등록
  • 릴리즈 차단 결함과 후속 개선 결함을 분리
  • 수정 후 동일 시나리오와 인접 회귀 범위를 재실행

5. 릴리즈 판정

  • 릴리즈 가능, 조건부 가능, 보류 중 하나로 판정하고 근거를 기록
  • 롤백 기준, 모니터링 지표, 장애 대응 Runbook 링크 확인
  • W5 테스트 결과 요약과 잔여 리스크를 HISTORY/TASK에 반영

완료 전 점검

  • W1~W4 기능 중 미검증 항목이 있는지 확인하고 audit 문서에 남긴다.
  • 실패 항목은 수정 완료, 릴리즈 보류, 후속 개선 중 하나로 분류한다.
  • CI 또는 수동 테스트 결과를 재현 가능한 형태로 남긴다.
  • Step 3 종료 시 릴리즈 가능 여부와 잔여 리스크를 팀 공통 문서에 반영한다.
