1. 먼저 이해할 것

이 작업은 새 기능을 늘리는 단계가 아니라 W1~W4에서 만든 기능이 실제 사용자 시나리오와 운영 조건에서 깨지지 않는지 확인하는 안정화 작업이다. 성공 케이스만 통과시키는 것이 아니라 실패 재현, 원인 기록, 수정 후 회귀 범위, 릴리즈 차단 여부를 분명히 남기는 것이 목표다.

따라서 W3 Step 7을 시작할 때는 제목의 기능명만 보고 바로 코드를 작성하지 말고, 원본 WORKFLOW의 요구사항 분석, Security 검토, DTO/Entity/API 설계, 테스트 항목을 먼저 연결해서 읽는다. 그 이해를 바탕으로 TASK 문서의 Instructions, Constraints, Output Format을 갱신한 다음 구현과 검증을 진행한다.

진행 기준

TASK 문서의 Step Goal, Done When, Scope, Input을 먼저 확인하고 PRD_W3 요구사항과 불일치하는 항목은 TASK Constraints에 반영한다. 구현 전 API, 이벤트, 저장 구조를 고정하고 테스트 통과 및 문서 반영까지 완료 기준으로 삼는다.

보안/권한 체크

  • Kafka 토픽 ACL 설정 확인 (서비스별 발행/소비 권한)
  • 이벤트 페이로드 민감정보 포함 여부 점검
  • 서비스 간 인증 토큰 전파 방식 확인
  • 결과 → TASK Constraints 반영
  • E2E 테스트 환경 시크릿 분리 확인
  • 테스트 데이터 민감정보 마스킹 확인
  • 테스트 환경 네트워크 격리 확인
  • 결과 → TASK Constraints 반영

설계 우선순위

  • 요청/응답, 저장 구조, 이벤트 형식을 구현 전에 고정한다.

1.1 TASK 시작

  • Step Goal / Done When / Scope / Input 확인
  • PRD_W3 해당 요구사항 확인 (서비스 간 통합 검증)
  • Duration 산정 확인

1.2 요구사항 분석

  • Kafka 이벤트 체인 E2E 시나리오 목록 도출
  • 서비스 간 이벤트 발행/소비 매핑표 작성
  • 코드 리뷰 승인 기준 정의 (PR 템플릿, 리뷰어 지정)
  • Instructions 초안 → TASK 문서 반영

1.3 Security 1차 검토

  • Kafka 토픽 ACL 설정 확인 (서비스별 발행/소비 권한)
  • 이벤트 페이로드 민감정보 포함 여부 점검
  • 서비스 간 인증 토큰 전파 방식 확인
  • 결과 → TASK Constraints 반영

1.4 E2E 테스트 시나리오 설계

  • gamification 이벤트 체인 시나리오 (카드 복습 → XP 적립 → 레벨업 → 배지 수여 → 알림)
  • community 이벤트 체인 시나리오 (신고 접수 → 모더레이션 → 알림)
  • card.review.due 이벤트 체인 시나리오 (스케줄러 → Kafka → 알림 발송)
  • audit 이벤트 소비 시나리오 (각 서비스 이벤트 → audit_logs 적재)
  • Duration(final) 갱신

1.5 Security 2차 검토

  • E2E 테스트 환경 시크릿 분리 확인
  • 테스트 데이터 민감정보 마스킹 확인
  • 테스트 환경 네트워크 격리 확인
  • 결과 → TASK Constraints 반영

1.6 테스트 데이터 준비

  • E2E 테스트용 시드 데이터 정의 (사용자, 카드, 노트, 커뮤니티 게시글)
  • Kafka 토픽 사전 생성 스크립트 작성
  • 테스트 실행 순서 의존성 정리

1.7 E2E 테스트 구현 및 조율

  • Kafka 이벤트 체인 E2E 테스트 작성 (Testcontainers)
  • 서비스 간 이벤트 발행 → 소비 → 결과 검증 자동화
  • 코드 리뷰 전 PR 승인 프로세스 적용
  • 각 서비스 담당자 테스트 결과 취합

1.8 통합 테스트 실행 및 검증

  • 전체 서비스 Docker Compose 기동 → E2E 테스트 실행
  • Kafka 이벤트 전파 지연 시간 측정 (< 3초 기준)
  • 이벤트 유실 여부 확인 (재시도 로직 검증)
  • 실패 테스트 원인 분석 및 담당자 배정

1.9 코드 리뷰 조율

  • 각 서비스 PR 리뷰 현황 취합
  • 크로스 서비스 영향도 리뷰 (이벤트 스키마 호환성)
  • PR 승인 및 main 브랜치 머지 조율

1.10 결과 정리

  • E2E 테스트 결과 리포트 작성
  • 미해결 이슈 목록화 및 우선순위 지정
  • RULE Reference → TASK 반영

완료 전 점검

  • 보안 1차/2차 검토 결과가 Constraints에 남아 있는지 확인한다.
  • 해당 단계에 필요한 단위, 슬라이스, 통합, E2E 테스트 통과 여부를 기록한다.
  • Swagger, 이벤트 스키마, 운영 가이드 등 문서 산출물이 있으면 RULE Reference에 연결한다.
